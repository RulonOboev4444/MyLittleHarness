# MyLittleHarness

MyLittleHarness is a portable, file-first harness for adding lightweight workflow memory, validation, repair, and read-only diagnostics to a target repository.

This repository contains the Python package, tests, reusable product docs, and minimal compatibility fixtures used to validate MyLittleHarness itself.

## What This Contains

- `README.md` as the product overview.
- `AGENTS.md` as the minimal operator contract for product work.
- `docs/` as the reusable MyLittleHarness product architecture/spec documentation spine.
- `.codex/project-workflow.toml`, `.agents/docmap.yaml`, `project/project-state.md`, and `project/specs/workflow/*.md` as minimal product compatibility fixtures for CLI/tests.
- `pyproject.toml`, `build_backend/`, `src/mylittleharness/`, and `tests/` for the 1.0.0 stdlib analyzer CLI, small primary command surface, compatibility command surface, hidden-help advanced diagnostics, explicit package-smoke verification, unified source-verified intelligence query search, read-only Git evidence suggestions, disposable projection helpers, bounded attach/repair mutation paths, create-only AGENTS and stable spec repair, marker-only detach apply, and optional read-only MCP projection adapter.

The compatibility fixtures are not working memory. They must not be used to open plans, preserve research, archive history, or operate the workflow from this tree.

The current productization posture is a local **1.0.0 release candidate**. The package version is `1.0.0`; the reusable docs, stdlib package posture, compatibility fixtures, and verification matrix describe one coherent product boundary. Release publication, package index upload, global installation, PATH or user-config mutation, and mutating workstation adoption remain separate operator decisions outside the v1 package contract. Standalone `bootstrap --apply` is not a product surface, and standalone switch-over automation is rejected.

## Analyzer CLI

This product tree includes a stdlib-first Python package for workflow reports and dry-run proposals.

The target product CLI is a small visible utility: `init`, `check`, `repair`, and `detach`. In 1.0.0, `init` and `check` are implemented primary commands, `repair` remains explicit with bounded AGENTS/state/docmap/stable-spec classes, `detach --dry-run` is a no-write preservation preview, and `detach --apply` is a marker-only disable path for eligible live operating roots. Advanced and compatibility commands remain available with command-specific help: `bootstrap --package-smoke` verifies local package install/import/console-script behavior from temporary locations, `intelligence --query` expands a single recovery query across exact/path/full-text search modes, `adapter --inspect --target mcp-read-projection` and `adapter --serve --target mcp-read-projection --transport stdio` provide optional read-only MCP projection access, and `closeout` can suggest read-only Git trailers from explicit closeout fields. Compatibility commands such as `attach`, `status`, and `validate` keep their existing flags and exit behavior. Advanced/compatibility commands are hidden or deemphasized in top-level help so first-contact usage stays focused on the primary command shape.

## First-Run Operator Path

Any file-reading, shell-capable agent can use MyLittleHarness from repo-visible files plus CLI reports. The portable operating-root start pass is to read `AGENTS.md`, `.codex/project-workflow.toml`, `project/project-state.md`, and `project/implementation-plan.md` only when `plan_status = "active"` or the user asks about plan/phase/closeout, then use `check` before mutating repair work. `.agents/docmap.yaml` routes docs reading when present, but routing metadata is not authority by itself.

The shortest safe first run is:

1. Use this product source checkout with `PYTHONPATH` pointing at `src`.
2. Optionally verify the local package and console script with `bootstrap --package-smoke`.
3. Point `--root` at the target repository and use the small command shape: `init`, `check`, `repair`, and `detach`.

For PowerShell, set `MLH_TARGET_ROOT` to the repository you want MyLittleHarness to inspect or attach:

```powershell
$ProductRoot = (Resolve-Path .).Path
$TargetRoot = $env:MLH_TARGET_ROOT
$env:PYTHONPATH = Join-Path $ProductRoot "src"
python -m mylittleharness --root $ProductRoot bootstrap --package-smoke
python -m mylittleharness --root $TargetRoot init --dry-run
python -m mylittleharness --root $TargetRoot check
python -m mylittleharness --root $TargetRoot repair --dry-run
python -m mylittleharness --root $TargetRoot detach --dry-run
```

Apply modes stay explicit and target-bound: `init --apply --project <name>`, selected `repair --apply` classes, and `detach --apply` should be run only after their dry-run output names the expected target repository and write boundary. Advanced diagnostics such as `status`, `validate`, `doctor`, `audit-links`, `tasks --inspect`, `bootstrap --inspect`, `intelligence`, `projection`, `semantic`, `adapter`, `snapshot`, `preflight`, `evidence`, and `closeout` remain available by explicit command for recovery, compatibility, and deeper review, but they are not required first-contact steps.

Codex skills, IDE-native rules, MCP clients, shell aliases, preflight wrappers, hooks, and CI may wrap this flow, but they are not correctness prerequisites and must not store the only copy of accepted decisions, current focus, docs decisions, repair approval, verification, or closeout evidence.

The package also declares the console script `mylittleharness`. The supported packaging smoke is `bootstrap --package-smoke`, performed from a temporary source copy into a temporary virtual environment with build artifacts kept outside the product source root. The package uses a small stdlib build backend from `build_backend/`, so no-network smoke checks do not require downloading or preinstalling a packaging backend. This smoke proves package metadata, importability, and the console script; it does not publish packages, change PATH, write user config, install hooks, add CI/GitHub workflows, switch roots, perform standalone bootstrap apply, or make packaging output authoritative.

## Install And Distribution

The supported local install story for this release candidate has three explicit paths:

- Source checkout usage: set `PYTHONPATH` to `src` and run `python -m mylittleharness` from this checkout for development or verification.
- Local package verification: run `python -m mylittleharness --root <product-checkout> bootstrap --package-smoke` from the product source checkout. This creates only temporary source, build, install, and virtual-environment locations outside the product root, performs a local no-network install, verifies `import mylittleharness` reports version `1.0.0`, and verifies the `mylittleharness` console script.
- Future distribution and adoption: `bootstrap --inspect` provides read-only adoption readiness evidence, including interpreter context, package metadata when run on the product checkout, and PATH discovery for the `mylittleharness` console script. Package-index upload, signed artifacts, global installation, PATH/profile changes, and user-config mutation are future-contract decisions. Standalone `bootstrap --apply` and standalone switch-over automation are rejected; any future adoption, publication, or migration checklist needs a scoped contract under its own command ownership. These decisions are not required for normal correctness and are not performed by the current CLI.

Local wheel artifacts may be built only in temporary or explicitly operator-selected output directories for verification. They must not be checked into the product source tree or treated as source of truth; the reusable authority remains `pyproject.toml`, `build_backend/`, `src/mylittleharness/`, tests, and product docs.

It refuses product-source compatibility fixtures, fallback/archive evidence, ambiguous roots, non-default state paths, malformed or partial state frontmatter, active-plan mismatches, symlink or file path conflicts, snapshot boundary conflicts, and non-selected content mutation; existing `AGENTS.md` files are never rewritten, lazy or not-required docmaps are skipped, and existing stable spec files are preserved. It reloads the target and runs post-repair `validate` plus `audit-links`; validation errors return `1`, while root or path authority failures return `2`.

The documented repair snapshot boundary is `.mylittleharness/snapshots/repair/` inside an explicit live operating root. Snapshots are pre-repair safety evidence with inspectable metadata, manual retention, and manual rollback instructions; they cannot approve repair, closeout, archive, commit, switch-over, or lifecycle decisions. Product-source fixtures, fallback/archive roots, ambiguous roots, generated-output roots, symlink/path conflicts, non-default state paths, missing docmap targets, AGENTS/stable-spec path conflicts, and snapshot boundary conflicts are reported as refusal or skip cases in the dry-run and apply paths and as warning posture in snapshot inspection. The implemented classes are snapshot-protected `project/project-state.md` frontmatter repair for prose-state operating roots, create-only `AGENTS.md` creation for missing required operator contracts, create-only `.agents/docmap.yaml` creation for missing required docmaps, create-only `project/specs/workflow/*.md` restoration for missing stable spec fixtures, and snapshot-protected `.agents/docmap.yaml` route repair for existing docmaps; malformed or partial state frontmatter, manifest normalization, active-plan repair, archive cleanup, AGENTS/stable-spec rewrites, and switch-over mutation remain out of scope.

## Product Readiness Contract

A MyLittleHarness release candidate is product-ready only when the required product surfaces are coherent: `README.md`, `AGENTS.md`, `docs/...`, `pyproject.toml`, `build_backend/`, `src/mylittleharness/`, `tests/`, and the minimal compatibility fixtures used by the CLI/tests. The current release-candidate posture satisfies that contract at package version `1.0.0`; release publication, package index upload, signed artifacts, global installation, mutating workstation adoption, and switch-over automation remain separate decisions. Standalone `bootstrap --apply` is not part of release-candidate readiness. The product source checkout must stay clean of operating memory, active plans, research intake, archives, generated validation artifacts, logs, caches, package debris, local databases, pycache, and repair snapshots.

The local release checklist is:

- package metadata and runtime version agree on `1.0.0`
- the stdlib build backend remains self-contained and declares no build-time dependency download requirement
- top-level help foregrounds only `init`, `check`, `repair`, and `detach`
- bytecode-disabled unit tests pass
- `validate`, `check`, `audit-links`, and `doctor` complete, with only expected advisory warnings
- `bootstrap --package-smoke` passes from temporary source/build/install locations outside the product source checkout
- product hygiene scan finds no bytecode, coverage files, build output, wheels, egg-info, generated validation reports, logs, local databases, active plans, research intake, archives, or repair snapshots in the product source checkout
- product-boundary wording keeps the shipped runtime model direct, rejects standalone `bootstrap --apply`, rejects standalone switch-over automation, and keeps publication, signing, global install, PATH/profile/user-config mutation, and workstation adoption outside normal correctness

Wheel, build, and install artifacts are verification outputs only unless a later publication plan accepts a durable artifact retention and signing policy.

Optional power-ups remain gated product capabilities. Semantic retrieval, generated evidence manifests, quality gates, adapter packs, hook/CI integration, package publishing, mutating workstation adoption, operating-root switch-over, additional repair classes, rollback automation, and snapshot cleanup require scoped contracts, tests, docs, degraded/offline behavior, and explicit non-authority wording before they can ship. Standalone `bootstrap --apply` is rejected rather than treated as an optional power-up. Generated evidence manifests are rejected as the default durable history path; Git trailers or commit metadata are the preferred durable evidence direction for Git repositories. Each optional helper must fail open to repo-visible files and must be safe to disable, delete, or ignore without losing accepted truth.

The default product dependency posture is a stdlib core with optional dependencies only through an explicit later policy. A dependency policy must name the install surface, offline behavior, generated-output boundary, provenance checks, and fallback path before any semantic runtime, provider adapter, package extra, publishing helper, mutating workstation adoption helper, or other external runtime becomes part of the product. Semantic runtime/provider selection is a decision point for the semantic retrieval phase, not a requirement of the current readiness baseline.

Release gates include bytecode-disabled unit tests, Core and operating-root smoke matrices, expected product-source apply refusals, docs and link audits, package smoke when packaging changes, adapter protocol smokes when adapter behavior changes, generated-output rebuild/delete checks when projection behavior changes, repair/snapshot evidence when mutation classes change, and hygiene scans for forbidden product-root residue. Closeout evidence belongs in the operating root and must record the observed `worktree_start_state`, `task_scope`, `docs_decision`, `state_writeback`, `verification`, `commit_decision`, residual risk, and carry-forward. For Git repositories, durable closeout history should prefer Git trailers or commit metadata assembled from those facts; for non-git roots, the fallback is repo-visible Markdown closeout fields and operator summaries. Generated projections, semantic results, evidence manifests, quality summaries, adapter state, hook or CI results, package artifacts, VCS state, and repair snapshots cannot approve closeout, archive, commit, repair, rollback, switch-over, or lifecycle decisions.

`docs_decision` values are `updated`, `not-needed`, and `uncertain`. A docs decision is required when behavior, CLI usage, configuration, setup, contract meaning, permissions, output shape, UX/copy, terminology, rollout, migration, or another user-facing meaning changes. `updated` means the affected reusable docs or operating-root docs were changed, `not-needed` means checked routed docs already match the change, and `uncertain` means the operator lacks enough evidence for confident closeout wording. `.agents/docmap.yaml`, `audit-links`, `check`, relevant specs, and observed diff evidence are the portable inputs; no Codex skill or generated docs-impact report is required for v1.

Finding severity is:
- `info` for successful measurements, optional/lazy surfaces, and advisory evidence.
- `warn` for reportable drift or budget/hygiene signals that do not block read-only reporting.
- `error` for required contract or product-posture failures that make `validate` fail.

Exit behavior:
- `validate` returns `1` when required-surface or consistency errors are found.
- `check` returns `1` when validation has error findings and otherwise returns `0`; warning-level status, validation, or drift findings remain advisory.
- `status`, `context-budget`, `audit-links`, `doctor`, `preflight`, `preflight --template git-pre-commit`, `tasks --inspect`, `bootstrap --inspect`, `bootstrap --package-smoke`, `semantic --inspect`, `semantic --evaluate`, `intelligence`, `evidence`, `closeout`, `adapter --inspect`, `adapter --serve`, `projection --inspect`, `projection --build`, `projection --delete`, `projection --rebuild`, `snapshot --inspect`, `init --dry-run`, `detach --dry-run`, `detach --apply`, `init --apply`, `attach --dry-run`, `attach --apply`, `repair --dry-run`, and `repair --apply` return `0` when the command completes successfully, even when advisory, hygiene, proposal, bootstrap-readiness, package-smoke success, stale-artifact, stale-index, semantic-readiness, semantic-evaluation degraded input, degraded-adapter-input, preflight warning/error, snapshot-inspection, missing-evidence, dirty-worktree, Git-unavailable, future-apply refusal posture, valid detach-marker idempotency, or idempotent no-op findings exist.
- CLI usage errors or unreadable target roots return `2`.
- `init --apply` and `attach --apply` return `2` when the target root is a product-source compatibility fixture, fallback/archive root, generated-output root, `--project` is required, or create-only path safety is ambiguous.
- `repair --apply` returns `1` when post-repair validation still has errors, and `2` when the target root or repair path is invalid or ambiguous.
- `detach --apply` returns `2` when the target root, authority paths, marker path, or existing marker payload is invalid or ambiguous.
- `bootstrap --package-smoke` returns `1` when the smoke cannot complete, package assertions fail, or the target is not the product source checkout.
