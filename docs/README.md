# MyLittleHarness Product Documentation

This directory is the reusable product documentation spine for MyLittleHarness.
It explains the product architecture and product-facing boundaries without treating any operating root as product authority.

Reusable product docs describe MyLittleHarness serving an explicitly targeted repository directly. The intended chain is `MyLittleHarness -> target repository`.

## Authority

These docs describe the reusable MyLittleHarness product architecture.
They do not turn the product repository into live workflow memory.

In shipped product behavior, the operating root is the user's target repository after MyLittleHarness is attached.

Legacy evidence is not a default input for fresh product-doc work. Open old material only for a named blocker.

This product tree may contain source, tests, the stdlib build backend, product docs, and minimal compatibility fixtures for CLI/tests.
It must not contain active plans, project memory, research/history/raw intake, archived plans, logs, caches, local databases, generated validation artifacts, or demo-workflow residue.

## Fixture Boundary

The workflow-shaped files under `.codex/`, `.agents/`, and `project/` in this product tree are compatibility fixtures for CLI reports and tests.
They are not the product architecture destination and are not operating memory.

Clean product architecture and specs belong under `docs/...`.

## Current Product Posture

MyLittleHarness is a solo-first, repo-native safety layer for AI-assisted development.
The product architecture keeps human-readable files as authority, uses metadata for routing, treats generated views as rebuildable projections, keeps analyzer reports separate from explicit apply mutation, defines repair snapshots as target-bound safety evidence for existing-content repair, and treats adapters as optional helpers that fail open to repo files.

The current productization posture is a local 1.0.0 release candidate at package version `1.0.0`. It is a verified product boundary for the direct `MyLittleHarness -> target repository` model, not a published package-index release, global installation flow, mutating workstation adoption step, or workstation mutation.

The target visible CLI is `init`, `check`, `repair`, and `detach`. The 1.0.0 implementation provides primary `init`, already-attached live-root no-op handling, `check`, no-write `detach --dry-run`, marker-only `detach --apply`, and bounded `repair` classes for state frontmatter, docmap metadata, docmap routes, and create-only stable workflow spec fixtures. `check` includes compact check-level drift for docmap gaps, stale root-role wording, primary instruction-surface size warnings, explicit delivered-vs-remainder token contradictions, live-root lifecycle route discovery, read-only route metadata warnings for malformed lifecycle frontmatter relationships, relationship hygiene findings under deep/focused hygiene diagnostics, product-source target artifact classification under link diagnostics, and report-only grain diagnostics for active-plan slice size, roadmap mapping/hygiene, and archived-plan calibration samples; `check --deep` and `check --focus validation|links|context|hygiene|grain` consolidate common read-only diagnostics while deeper section-size details remain advanced diagnostics. Product docs and stable specs are measured as informational context detail rather than routine warning pressure.

It also keeps explicit `bootstrap --package-smoke` package verification, unified `intelligence --query` recovery search with disposable navigation-cache refresh inside `.mylittleharness/generated/projection/`, route-only `intelligence --focus routes`, optional read-only MCP projection access through `adapter`, read-only Git trailer suggestions to `closeout`, explicit future-idea incubation through hidden `incubate --dry-run|--apply --topic <topic> --note <note>` or `--note-file <path>` plus `--fix-candidate` for standard MLH debt capture, deterministic active-plan scaffolding and optional narrowed roadmap item association through hidden `plan --dry-run|--apply --title <title> --objective <objective> [--roadmap-item <id>] [--only-requested-item]`, reviewed closeout/archive/next-plan orchestration through hidden `transition --dry-run|--apply`, explicit research/incubation lifecycle cleanup plus read-only relationship scanning, text-input audit posture, Entry Coverage diagnostics, terminal `--archive-covered` archival, and split suggestions through hidden `memory-hygiene --dry-run|--apply` and `memory-hygiene --dry-run --scan`, explicit accepted-work roadmap item mutation plus reciprocal source-incubation metadata through hidden `roadmap --dry-run|--apply --action add|update --item-id <id>`, explicit closeout/state synchronization, optional active-plan or project-state closeout fact harvesting, structured `product_source_root` recording, optional roadmap/source-incubation closeout sync, coverage-aware incubation auto-archive, completed-plan archiving, and post-writeback plus compact-only operating-memory compaction through hidden `writeback --dry-run|--apply`, compatibility names such as `attach`, `status`, and `validate`, and advanced diagnostics including `doctor`, `preflight`, `context-budget`, `audit-links`, `intelligence`, `projection`, `semantic`, `evidence`, `closeout`, `snapshot`, `adapter`, `bootstrap`, and `tasks`. Those diagnostics remain read-only or explicitly bounded where implemented, but they are progressive-disclosure surfaces rather than the product front door. Top-level help foregrounds only `init`, `check`, `repair`, and `detach`; advanced/compatibility command-specific help and normal report stdout stay available and compatible.

The first-run operator path is deliberately shorter than the full diagnostic surface: run from the product source checkout with `PYTHONPATH=src`, optionally run `bootstrap --package-smoke` against the product checkout, then point `--root` at the target repository and start with `init --dry-run`, `check`, `repair --dry-run`, and `detach --dry-run`. Apply modes stay explicit and target-bound after dry-run review. `bootstrap --inspect`, `tasks --inspect`, publication, global installation, PATH/profile edits, user-config mutation, hooks, CI, MCP clients, semantic providers, and workstation adoption are not prerequisites for first-run correctness.

The operating-root start pass is portable across agents that can read files and run shell commands. It starts from `AGENTS.md`, `.codex/project-workflow.toml`, `project/project-state.md`, and the active plan only when `plan_status = "active"` or the user asks about plan/phase/closeout. When a plan is active, `active_phase` and `phase_status` in `project/project-state.md` are the first-class continuation pointers; CLI reports warn when they are missing so agents do not have to infer phase state from recent prose. MLH write rails also refresh marked Current Focus and Memory Routing Roadmap hot-pointer blocks from lifecycle frontmatter when plans open, phases move, plans close, or a reviewed transition composes those steps. Live-root `status`/`check` summarize whether the active phase still has implementation work pending or whether the next action is explicit closeout/writeback, archive, transition review, or manual commit.

Route discovery is part of the visible lifecycle routing contract: live-root `status`/`check` and `intelligence --focus routes` name the canonical state, active-plan, optional roadmap, incubation, research, stable-spec, decision/do-not-revisit, ADR, verification through the active-plan block and optional `project/verification/*.md` proof/evidence records, closeout/writeback, archive, and docs-routing lanes without turning `AGENTS.md` into a long playbook. `project/roadmap.md` is an optional live-root sequencing surface for accepted work between raw incubation and one bounded active implementation plan; it is read when planning or sequencing requires it, not as mandatory start-path material. The hidden `roadmap --dry-run|--apply` rail is the mechanical write path for that roadmap lane: it requires an existing managed `project/roadmap.md`, adds or updates one item block from explicit fields, preserves text outside the target item block, refuses unsafe roots, malformed roadmaps, duplicate ids, invalid statuses, bad relationship paths, and missing related item ids, and writes only `project/roadmap.md`.

`plan --roadmap-item` and `writeback --roadmap-item` reuse the same bounded item-block validation to sync explicit plan relationships or same-request closeout summaries without turning roadmap output into approval. Plan opening may batch the requested roadmap item plus roadmap-slice `covered_roadmap_items`, `plan --only-requested-item` narrows that opening to one requested item, and archive closeout may batch the requested roadmap item plus active-plan `covered_roadmap_items` in one writeback apply. Live-root `check`/`validate` also report `route-metadata-*` warnings when routed roadmap, research, incubation, stable-spec, decision, ADR, or verification artifacts carry malformed status, unsafe relationship paths, stale active references, or missing archive targets; those warnings are advisory only.

`incubate --dry-run|--apply` is the hidden mechanical write rail for that incubation lane: it accepts explicit topic/note fields or UTF-8 `--note-file` input for multi-paragraph text, can add `[MLH-Fix-Candidate]` with `--fix-candidate`, creates the canonical directory when safe, creates or appends the same safe-slug note, reports note body line/character/hash identity, and does not decide whether the idea is good or infer missing meaning from arbitrary text. `plan --dry-run|--apply` is the hidden mechanical write rail for the active-plan lane: it accepts explicit title/objective/task fields, creates the default implementation-plan scaffold, updates lifecycle frontmatter and the Current Focus block, refuses unsafe roots, invalid roadmap links, or active-plan conflicts before writing, and does not infer product architecture or act as an LLM planner. `writeback --from-active-plan` can harvest explicit closeout bullets from active-plan Closeout Summary/Facts/Fields sections before archive closeout, falling back to matching project-state closeout authority when the plan has no explicit section; `writeback --product-source-root` records validated product-source metadata. `transition --dry-run|--apply` is the hidden reviewed orchestration rail for explicit phase completion, active-plan archive, and next-plan opening; apply requires the dry-run review token and stops on token drift before writes. `memory-hygiene --dry-run|--apply` is the hidden mechanical lifecycle cleanup rail for MLH-owned research/incubation artifacts: it accepts explicit source, promoted destination, archive destination, status, Entry Coverage, archive-covered, and exact-link repair fields; refuses unsafe roots, product fixtures, path escapes, symlinks, conflicts, and ambiguous source routes before writing; and does not infer semantic promotion meaning. Product-source fixtures do not emit live route-table rows, and route output remains advisory rather than authority for repair, closeout, archive, commit, or lifecycle decisions. `check`, `status`, `validate`, `audit-links`, `evidence`, and `closeout` are CLI reports over those files; they do not require Codex skills, IDE-native skills, MCP clients, hooks, CI, or workstation adoption.

Roadmap-backed plan opening now has a visible slice contract. The roadmap side can store advisory `execution_slice`, `slice_goal`, `slice_members`, `slice_dependencies`, `slice_closeout_boundary`, `source_research`, `related_specs`, and `target_artifacts` fields on managed item blocks. The plan side records the executable one-slice contract in active-plan frontmatter with `primary_roadmap_item`, `covered_roadmap_items`, `domain_context`, `target_artifacts`, `execution_policy`, `auto_continue`, `stop_conditions`, and `closeout_boundary`. Generated plans default to current-phase-only execution with `auto_continue = false`; the operator continues only the active phase named by `project/project-state.md`, records repo-visible evidence/state, and then stops. Any opt-in auto-continuation must be explicit and must name stop-condition coverage for verification, authority/docs impact, write scope, source reality, sensitive action, and closeout boundaries. These fields help execution and relationship projection, but they do not approve lifecycle, closeout, archive, repair, commit, rollback, or next-slice movement.

When a live operating root records product-source paths in `target_artifacts` or product-source context, link diagnostics classify those references as `product-target-artifact` instead of requiring product files to exist inside the operating root. The in-memory relationship projection uses the parallel `product-target` status for target-artifact edges.

For roadmap-backed plans, MyLittleHarness can also emit a `Plan Synthesis Notes` section plus dry-run/apply findings for bundle rationale, split boundary, target-artifact pressure, and phase pressure. This makes bundle/split decisions inspectable without turning roadmap output into a hard sizing gate or lifecycle decision.

The current install and distribution story is local and bounded. Operators can run from a source checkout with `PYTHONPATH=src`, or verify the package path with `bootstrap --package-smoke`, which performs a temporary no-network install outside the product root and proves metadata, import/version, and the `mylittleharness` console script. `bootstrap --inspect` provides no-write adoption readiness evidence, including interpreter context, package metadata when run on the product checkout, and PATH discovery for the console script. Package-index publication, signed artifacts, global installation, PATH or profile edits, and user-config mutation remain separate future decisions, not prerequisites for normal correctness. Standalone `bootstrap --apply` and standalone workstation mutation are rejected as product surfaces.

Durable evidence direction is Git-native: Git trailers or commit metadata for Git repositories, and repo-visible Markdown closeout fields plus operator summaries for non-git roots. Read-only `closeout` reports may suggest paste-ready Git trailers from explicit closeout fields, and `evidence` reports may assemble source-bound candidates, but neither stages, commits, archives, repairs, approves lifecycle decisions, or creates persistent generated evidence stores.

Archive-active-plan is a completed-lifecycle close rail: it refuses pending `phase_status` before any archive, roadmap, or source-incubation write, and the archived plan keeps synchronized completed `status`/`phase_status` derived copies.

Docs decisions are explicit closeout facts, not skill-owned side effects. The accepted values are `updated`, `not-needed`, and `uncertain`; `uncertain` keeps closeout language provisional. A docs decision is required when behavior, CLI usage, configuration, setup, contract meaning, permissions, output shape, UX/copy, terminology, rollout, migration, or other user-facing meaning changes. The marked MLH closeout writeback block in `project/project-state.md` is the current closeout fact authority for MLH-owned state writeback. New closeout writeback blocks record plan identity fields such as `plan_id`, `active_plan`, and `archived_plan`; dry-run/apply reports whether existing facts will carry, replace, or refuse before a partial closeout can mix facts from another plan. Active-plan frontmatter fields and exact active-plan closeout bullets inside explicit closeout summary/facts/fields sections are derived diagnostic copies; schema examples and closeout checklists are ignored by closeout sync. Lifecycle-only phase-pointer writeback does not copy prior closeout facts into the active plan. `writeback --from-active-plan` turns exact active-plan closeout bullets, or complete matching project-state closeout authority facts when the active plan has no explicit closeout section, into same-request closeout facts before preflight, with explicit CLI fields taking precedence over harvested values. `writeback --apply` synchronizes requested derived copies with the state writeback block, maps written `phase_status` to an existing active-plan frontmatter `status` copy, updates the matching active-phase body `- status:` line, records validated `product_source_root` when requested, and refreshes marked Current Focus and Memory Routing Roadmap hot-pointer blocks from lifecycle frontmatter; harvested facts are treated as requested facts after preflight. Optional `--roadmap-item` sync uses only same-request closeout facts plus the active or archived plan path. Lifecycle `phase_status = complete` becomes `done` in the phase body. `writeback --apply --phase-status complete` records a ready-for-closeout boundary and keeps the active plan open; it does not archive the plan, default roadmap items to `done`, archive source incubation, or open the next slice. `writeback --apply --archive-active-plan` is the explicit live-root close rail for the default active plan: it moves the plan to `project/archive/plans/`, clears active lifecycle pointers, records `last_archived_plan`, and refreshes Current Focus while preserving supplied or harvested closeout facts. `transition --dry-run|--apply` can compose phase completion, active-plan archive, and next-plan opening only after dry-run review; apply requires the matching review token and stops on token drift before writes, without staging or committing. After successful live-root `writeback --apply` and after safe plan lifecycle writes, oversized `project/project-state.md` files above the default 250-line threshold may be compacted automatically into a reference archive under `project/archive/reference/`; `writeback --dry-run|--apply --compact-only` runs only that compaction rail without changing closeout facts, lifecycle frontmatter, active-plan copies, or roadmap fields. Dry-run reports whether compaction would run, skip, or refuse, and refusal does not undo a safe state writeback. `.agents/docmap.yaml`, `audit-links`, `check`, relevant specs, and observed diff evidence are the portable inputs.

The first owned generated-output boundary is `.mylittleharness/generated/projection/`. It may contain rebuildable JSON projection artifacts, a rebuildable `relationships.json` graph derived from repo-visible relationship metadata, and the disposable SQLite FTS/BM25 index, including relaxed plain-text recovery search over source-verified rows; deleting it must not change accepted project truth. `intelligence --path`, `--full-text`, and `--query` may refresh missing or stale projection artifacts/indexes inside this boundary when navigation needs them, while `check`, `status`, route reports, semantic readiness/evaluation, adapter inspection/stdio serving, optional preflight templates, snapshot inspection, and bootstrap/package readiness keep their no-write or explicit-boundary contracts. `bootstrap --inspect` is the no-write workstation adoption readiness surface; it can report context but cannot adopt, install, or mutate workstation state. `bootstrap --package-smoke` verifies local install/import/console-script behavior only from temporary locations outside the product source root. Future semantic retrieval runtime, generated semantic output boundaries, generated evidence manifests, additional adapter families, publishing automation, mutation-capable VCS/GitHub behavior, additional hook/CI integrations, and additional mutation procedures require later scoped plans before implementation. Future adoption or publication behavior must use a scoped contract under its own command ownership, not a generic bootstrap apply lane.

`detach --dry-run` reports root posture, preserved authority files, marker target, generated projection posture, manual recovery notes, and boundary reminders without writing files. `detach --apply` creates only `.mylittleharness/detach/disabled.json` in eligible live operating roots, leaves valid existing markers unchanged, treats generated projection artifacts as disposable but preserved, and reports product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, non-default authority paths, unreadable manifest/state surfaces, path conflicts, and invalid marker payloads as fail-closed apply inputs. `disable` is explanatory terminology only, not a command spelling.

The repair snapshot boundary is `.mylittleharness/snapshots/repair/` inside an explicit live operating root. Snapshots preserve pre-repair bytes and metadata for inspection only; they cannot approve repair, closeout, archive, commit, lifecycle decisions. `snapshot --inspect` is the terminal-only no-write inspection surface for snapshot presence, metadata readability, copied-file/hash/path posture, current-target posture, retention, manual rollback instructions, and non-authority wording. `repair --dry-run` reports a no-write `state-frontmatter-repair` snapshot plan only for default-path prose `project/project-state.md`, reports no-write create-only plans for a missing required `AGENTS.md`, missing required `.agents/docmap.yaml`, or missing `project/specs/workflow/*.md` stable spec fixtures, and reports the no-write `.agents/docmap.yaml` route-repair snapshot plan when existing docmap route diagnostics exist. `repair --apply` can snapshot and prepend deterministic state frontmatter before stopping, creates absent required AGENTS contracts, docmaps, and stable spec fixtures without snapshots, and creates a timestamped snapshot before adding missing route entries to an existing docmap. Product fixtures and legacy reference material remain outside state, AGENTS, docmap, stable-spec, and snapshot creation.

Product readiness means the reusable source tree, docs, tests, package metadata, stdlib build backend, and compatibility fixtures are coherent while operating memory and closeout evidence stay in an explicit operating root. The 1.0.0 baseline is the small visible CLI, stdlib package posture, local package-smoke verification, explicit repair/detach mutation boundaries, source-verified intelligence/evidence helpers, optional read-only MCP projection access, disposable generated projections, and gated optional power-ups. The current release checklist is documentation-and-verification based: package metadata/runtime version agreement, bytecode-disabled tests, read-only health gates, package smoke from temporary locations outside the product root, product hygiene, and product-boundary wording. Optional power-ups become release candidates only after their contracts name source-of-truth placement, generated or adapter boundaries, dependency policy, degraded/offline behavior, verification, closeout evidence, and non-authority rules. The stdlib core remains the default dependency posture; semantic runtimes, provider adapters, package extras, publishing helpers, mutating workstation adoption helpers, and other external runtimes require later scoped policy before they become product surfaces. Package-index publication, signed artifact release, global installation, PATH/profile/user-config mutation, and workstation adoption are not part of local release-candidate correctness; standalone `bootstrap --apply` and standalone workstation mutation are rejected.

For a clean operating-project pilot, `architecture/clean-room-carry-forward.md` carries the reusable invariants and risk register without importing unrelated historical context.
