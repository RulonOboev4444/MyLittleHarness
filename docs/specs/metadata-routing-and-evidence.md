# Metadata Routing And Evidence Spec

## Purpose

This spec defines how MyLittleHarness uses metadata, routing surfaces, Git-native evidence, and doc-gardening checks without making them stronger than file-visible authority.

Metadata should help agents and tools find the right files, understand lifecycle, and preserve provenance. It must not become the only meaning of a document.

## Authority

Human-readable repo files remain the recovery baseline. Stable product contracts live under `docs/...` in the product root. Active operating-project focus, plan status, and research intake live in the operating root for the repository MyLittleHarness is servicing.

Metadata may route to authority, classify lifecycle, point at source sets, name provenance, and support validation. The surrounding Markdown content must remain meaningful when metadata is missing, stale, or partially invalid.

Evidence has authority only when it points back to inspectable files, commands, observations, accepted docs, or commit metadata. Evidence paperwork alone is not completion.

## Non-Authority

The following are advisory or generated support, not authority:

- YAML frontmatter by itself
- docmaps and routing maps
- persistent generated evidence manifests
- repair snapshot manifests
- backlink or link reports
- stale-pointer reports
- quality-gate summaries
- search results and metadata indexes
- adapter memory from skills, MCP, browser, IDE, GitHub, CI, hooks, or task runners

Metadata cannot authorize mutation, closeout, repair, or archiving without the matching repo-visible plan and validation evidence. Repair snapshot metadata is included in this rule even when it records copied file hashes and rollback instructions.

## Current Contract

Metadata and routing are allowed when they reduce context load and improve recovery. They must stay lightweight, repo-visible when durable, and subordinate to the files they describe.

Target repository work keeps active memory and new research in that target repository. The product repository receives reusable product specs only. Legacy reference material is not part of default routing.

Routing surfaces should behave like tables of contents or impact maps, not hidden policy engines. Weak semantic matches, stale docmaps, and incomplete metadata may guide reading, but they must not authorize writes.

## V2 Governance Metadata

The v2 foundation promotes agent-governance metadata as protocol data, not as a hidden control plane. The first durable contract is to make future reports, manifests, run records, claims, approval packets, and drift proposals source-bound and reviewable.

Structured route and finding metadata should be able to name:

- `route_id`
- `authority_class`
- `mutates`
- `requires_human_gate`
- `gate_class`
- `human_gate_reason`
- `allowed_decisions`
- `suggested_action`
- `advisory_only`

Agent run evidence should be able to name:

- `run_id`
- `parent_run_id`
- `role_id`
- `actor_id`
- `task_id`
- `input_refs`
- `input_hashes`
- `context_tier_refs`
- `output_refs`
- `claimed_paths`
- `commands`
- `verification_refs`
- `status`
- `stop_reason`
- `attempt_budget`
- `attempt_count`
- `repeated_failure_signature`

When known, evidence may also record model/tool provenance such as `provider`, `model_id`, `adapter`, `tool_schema_hash`, and `output_schema_hash`. These fields are evidence about how output was produced. They do not make MLH a provider gateway and do not make provider metadata authoritative.

Agent run records are the first concrete v2 evidence record. The MVP route is `project/verification/agent-runs/*.md`, written only through explicit `evidence --record --dry-run|--apply` in live operating roots. Records must name role, actor, task, input refs, output refs, claimed paths, commands, status, stop reason, attempt budget, optional repeated-failure signature, optional provider/model/tool provenance, and source hashes for referenced local files. They are cold evidence inputs, not hot state, and they cannot approve lifecycle transitions, archive, roadmap status, staging, commit, rollback, hidden worker spawning, or next-plan opening. Bare `check` reads them without writing and warns when required metadata is malformed or recorded source hashes have gone stale.

Work claims should be repo-visible before broad fan-out:

- `claim_id`
- `run_id`
- `role_id`
- `owned_paths`
- `read_only_paths`
- `expires_at`
- `priority`
- `conflict_policy`
- `release_condition`

Review tokens and approval packets should bind approval to planned writes and verification, not to an agent's self-report. They should preserve parent operation identity so a risky operation cannot bypass review by being split into many smaller patches.

Reconcile output is proposal metadata only. It may classify a spec/code/evidence relation as `unassessed`, `partially_verified`, `synced`, `stale`, or `drift_detected`, and it may propose amendment, supersession, verification-needed, or drift-record artifacts. It must not silently edit accepted specs to match implementation.

## Intake Routing

Incoming information may be classified before it becomes incubation clutter. Hidden `intake --dry-run` accepts explicit text or UTF-8 `--text-file` input and reports one route advisor classification for future ideas, research imports, decisions, ADRs, verification evidence, docs impact, archive references, or ambiguous input. The report is advisory only.

`intake --apply` is intentionally narrower than the classifier. It requires a live operating root, unambiguous classification, and an explicit new root-relative Markdown `--target` whose route matches the classifier. It refuses product fixtures, fallback/archive roots, path escapes, symlinks, existing targets, ambiguous text, and mismatched destinations before writing. Apply writes only that one target file and cannot approve roadmap promotion, repair, closeout, archive, commit, rollback, or lifecycle decisions.

## Relationship Hygiene

Lifecycle relationship hygiene uses a minimal path/id vocabulary rather than a separate authoritative index. Accepted fields include provenance links such as `source_incubation`, `related_incubation`, and `source_research`, roadmap links such as `related_roadmap` and `related_roadmap_item`, plan links such as `related_plan`, `archived_plan`, and `implemented_by`, supporting links such as `related_spec`, `related_decision`, `related_adr`, and `related_verification`, closeout facts such as `verification_summary`, `docs_decision`, and `carry_forward`, archive and promotion links such as `archived_to` and `promoted_to`, and lifecycle relation fields such as `supersedes`, `superseded_by`, `merged_into`, `merged_from`, `split_from`, `split_to`, `rejected_by`, and `rejection_reason`.

Read-only relationship hygiene can be surfaced through `check --focus hygiene`, `check --deep`, or `memory-hygiene --dry-run --scan`. It may report stale relationship paths, missing reciprocal roadmap/source-incubation links, orphan incubation notes, done roadmap items without archive or verification links, implemented notes still in active incubation, coverage-aware archive candidates, Entry Coverage blockers, incubation cleanup advisor classifications for keep-active notes, follow-up extraction, archive candidates, exact-link repair candidates, and ambiguous notes, CLI text-input audit posture, and heuristic split suggestions. These findings are diagnostics only.

Meta-feedback capture is operating-memory evidence, not lifecycle authority. The `meta-feedback` rail writes canonical incubation notes only in the central MyLittleHarness-dev live operating root, using `--to-root <central-root>`, `MYLITTLEHARNESS_META_FEEDBACK_ROOT`, or `--root` when it already points at that central root; noncentral serviced or product roots remain observed-root provenance and are refused as canonical MLH product-debt destinations. Canonical notes carry managed cluster metadata such as `canonical_id`, `friction_signature`, `occurrence_count`, `first_seen`, `last_seen`, `observed_roots`, `affected_routes`, `duplicate_topics`, `recurrence_score`, representative examples, and `latest_observation_hash`; exact-signature duplicates can append automatically and `--dedupe-to <canonical-id>` supports explicit near-duplicate appends. Matching accepted roadmap items may be added or updated with recurrence context in the same central root, but roadmap placement remains sequencing evidence only and cannot approve repair, closeout, archive, lifecycle movement, release cleanup, staging, commit, push, or next-plan opening. Apply reports include post-apply lifecycle posture evidence: if `plan_status`, `active_plan`, phase fields, or active-plan file posture move during the command, the report warns with before/after values and routes the operator back to `check` and repo-visible route inspection instead of silently treating the movement as approved.

Same-transaction writeback is allowed only for facts structurally known from the requested operation. `incubate --apply` may record the current active plan relationship, and the active plan's roadmap item when present, in newly created incubation notes; appending to an already-related note preserves its existing relationship metadata. `roadmap --apply --source-incubation <rel>` may update an unowned source note with reciprocal roadmap fields; if the source note already has owner-like roadmap, plan, promotion, archive, or implementation metadata, the roadmap item records non-owning `related_incubation` instead and does not retarget the note. `plan --roadmap-item <id>` records the related roadmap item and source incubation in the active-plan frontmatter when available, carries non-owning `related_incubation` into active-plan metadata and roadmap-derived candidate context without retargeting that source note, syncs `related_plan = project/implementation-plan.md` to the requested roadmap item plus roadmap-slice `covered_roadmap_items`, and refreshes each explicit roadmap source-incubation note so its `related_plan` points at the new active plan. `writeback --roadmap-item <id>` may sync same-request closeout facts to the requested roadmap item, active-plan `covered_roadmap_items`, and the explicit source incubation. These relationship-sync paths accept canonical item blocks and existing compatibility top-level `## RM-...` roadmap sections; legacy `slice_members` labels that are not roadmap ids are left as work-area metadata. When a sync owns a source-note relationship field, it replaces the whole targeted scalar block, including stale YAML list continuations that would otherwise leave malformed frontmatter; this relationship frontmatter block recovery is bounded to those explicit fields and does not make route metadata diagnostics a repair proposal.

Whole-file incubation auto-archive is intentionally narrow. It runs only after a requested active-plan closeout/archive operation proves that a single-entry source incubation note is fully covered by a done roadmap item, archived plan, verification summary, and final docs decision. The implemented archive lane is `project/archive/reference/incubation/**`. If an explicit `source_incubation` already points into that archive lane, writeback may update same-request relationship frontmatter in place, but it must not derive a second archive target, delete the existing archived note, or repair links as though the note moved. Mixed notes, notes with `open_threads` or `contains`, multiple dated entries, open-question/follow-up/deferred headings, or unchecked task items remain active and receive blocker findings until a later entry-coverage model can prove every live thread has a destination.

Evidence should be captured when provenance value exceeds ceremony cost, especially for verification, closeout, repair, or product-root hygiene. Small ad hoc work may rely on a concise chat summary and project-state writeback when required.

For Git repositories, durable closeout history should prefer commit metadata such as Git trailers. Suggested trailer fields may be assembled from repo-visible closeout facts, but MyLittleHarness helpers must not stage files, create commits, amend commits, push, mutate Git config, or decide lifecycle state. For non-git roots, the durable fallback is repo-visible Markdown closeout fields and operator summaries. Persistent generated evidence manifests are rejected as the default fallback.

Docs decisions are portable closeout facts. Use `docs_decision = updated` when the relevant docs were changed, `docs_decision = not-needed` when routed docs were checked and already match the work, and `docs_decision = uncertain` when the operator lacks enough evidence to state either result. `uncertain` blocks confident closeout language until the uncertainty is resolved or explicitly carried forward. The current MLH-owned closeout authority is the marked closeout writeback block in `project/project-state.md`; `active_phase`, `phase_status`, `last_archived_plan`, and optional `product_source_root` remain frontmatter authority in `project/project-state.md`. New closeout writeback blocks record plan identity metadata (`plan_id`, `active_plan`, and `archived_plan`) before closeout fields. Partial closeout updates may carry existing facts only when the recorded identity matches the current active or archived plan; when identity is absent or different, writeback refuses the partial update and requires complete replacement facts. Complete same-request closeout facts may replace stale or legacy facts without carrying them. The bounded exception is a phase-only ready-for-closeout handoff: a same-request `docs_decision = uncertain` plus `phase_status = complete` may replace mismatched prior facts with the current plan identity and provisional docs decision, without carrying stale closeout fields or archiving the plan. Active-plan frontmatter fields, exact active-plan closeout bullets inside explicit Docs Decision, State Transfer, or closeout summary/facts/fields sections, and the active-phase body status line are derived diagnostic copies; schema examples, roadmap fields, and closeout checklist items are not derived closeout copies. The marked Current Focus summary and Memory Routing Roadmap hot pointer are derived from lifecycle frontmatter. When an active implementation plan carries `docs_decision` in frontmatter, `validate` reports unsupported values and reports `uncertain` as stale closeout posture when `project/project-state.md` says `phase_status = complete`. When project-state says `phase_status = complete` but an active-plan frontmatter `status` copy still says `active`, or when the matching active phase body still says `pending` or `in_progress`, `check`/`validate` report lifecycle drift instead of forcing agents to infer the split from prose history. If completed closeout facts are already present while the default active plan remains active, `check`/`validate` report ready-for-closeout boundary posture rather than treating the still-open plan as automatic archive drift.

The managed Current Focus summary and Memory Routing Roadmap hot pointer are derived labels over lifecycle frontmatter. Existing MLH-owned roadmap `Current Posture` sections are derived labels over roadmap item metadata and may be refreshed by roadmap item writes or plan/writeback relationship sync. They label older prose/archive history as reference context, and live-root `check` reports `state-lifecycle-prose-drift` when current non-historical project-state prose appears to disagree with `plan_status`; rewrite or compact-only action stays explicit.

A docs decision is required when behavior, CLI usage, configuration, setup, contract meaning, permissions, output shape, UX/copy, terminology, rollout, migration, or other user-facing meaning changes. Portable inputs are the actual file diff or task evidence, `.agents/docmap.yaml` when present, `audit-links`, `check`, relevant product specs, and explicit closeout fields. A Codex skill, generated docs-impact report, IDE rule, MCP client, hook, or CI result may help route attention, but it cannot be required for the decision and cannot store the only copy of the decision.

The first implemented evidence helper is the read-only `evidence` CLI report. It collects candidate verification anchors, closeout fields, validation sections as verification closeout candidates, durable proof/evidence records under optional `project/verification/*.md`, agent run records under optional `project/verification/agent-runs/*.md`, residual risks, skip rationale, and carry-forward cues from inventory-discovered source files and currently readable workflow surfaces. Candidate cue identity is report-only and derived from cue kind, source path, line number, normalized preview text, and a deterministic hash. Concrete closeout fields are limited to explicit field bullets, exact field headings, or observed result lines; manifest language, future-contract wording, durable proof/evidence records, and agent run records remain context until an explicit closeout field records how they were used. It is terminal-only, creates no persistent evidence manifest, report file, generated output, database, cache, adapter state, quality-gate state, or quality-gate record, and cannot authorize mutation, closeout, archive, commit, plan lifecycle changes.

The first closeout assembly helper is the read-only `closeout` CLI report. It combines source-bound evidence cues, durable proof/evidence record prompts, manifest policy, read-only Git evidence suggestions, report-only quality/readiness cues, projection posture, and a fail-open VCS probe into operator-facing closeout prompts. The VCS probe may run only target-bound read-only Git discovery and porcelain status commands; non-git roots, missing Git, subprocess failure, clean status, and dirty status all remain report findings rather than authority. Concrete closeout field lines are explicit field bullets, exact field headings, observed result lines, or fields in the MLH closeout writeback block. Broad future-contract mentions and durable proof/evidence records remain context when concrete field evidence is absent. For Git worktrees, the report may suggest trailers for `worktree_start_state`, `task_scope`, `docs_decision`, `state_writeback`, `verification`, `commit_decision`, residual risk, and carry-forward from explicit repo-visible closeout lines, preferring the current state writeback block over stale derived plan copies. For non-git or unknown Git posture, it reports the Markdown/operator-summary fallback and emits no trailer suggestions. It cannot decide task scope, approve completion, archive plans, stage files, commit, repair, write persistent evidence manifests, write quality-gate state, or write lifecycle state.

The implemented mutating closeout/state path is hidden `writeback --dry-run|--apply`. Dry-run is no-write. Apply is allowed only for a live operating root with readable frontmatter state, refuses product-source fixtures, fallback/archive roots, ambiguous roots, malformed frontmatter, and unsafe active-plan targets, and writes only the MLH closeout writeback block in `project/project-state.md`, requested lifecycle/product-source frontmatter fields in `project/project-state.md`, the marked Current Focus block and Memory Routing Roadmap hot pointer derived from frontmatter, matching derived active-plan frontmatter/body fields when those closeout copies already exist in explicit Docs Decision, State Transfer, or closeout summary/facts/fields sections, requested roadmap item plus active-plan covered-item relationship/fact sync, structurally known source-incubation relationship metadata, explicit archive retargeting for MLH route frontmatter that pointed at the active plan, the canonical active-plan archive move when `--archive-active-plan` is requested, coverage-aware source-incubation auto-archive during archive close, and operating-memory compaction when safe. Reviewed `transition --archive-active-plan --next-roadmap-item` supplies the next item's source-incubation note as an archive-retarget skip so the next plan's source does not get rewritten to the just-archived plan before plan opening. It synchronizes one-line `worktree_start_state`, `task_scope`, `docs_decision`, `state_writeback`, `verification`, `commit_decision`, `residual_risk`, and `carry_forward`, refusing multiline closeout field input before writes; it also writes requested `active_phase`, `phase_status`, `last_archived_plan`, and validated `product_source_root` frontmatter fields. With `--from-active-plan`, explicit active-plan closeout sections are preferred, while complete matching project-state closeout authority facts are an allowed fallback. Closeout dry-run/apply reports the current plan identity and whether existing closeout facts will be carried, replaced, or refused. Lifecycle-only phase-pointer writeback does not carry previous closeout facts from `project-state` into the active plan; if the only closeout fact is `docs_decision = uncertain` and the lifecycle write sets `phase_status = complete`, mismatched previous facts are replaced with the current plan identity and the provisional docs decision instead of being carried. Optional `--roadmap-item <id>` sync uses only selected same-request closeout values plus the active or archived plan path; it may include active-plan `covered_roadmap_items`, and it does not copy carried project-state closeout values into roadmap fields. When the roadmap item names an explicit `source_incubation`, archive close may also update that source note from the same-request facts and archive it only when structural coverage is proven: single-entry notes can be covered by the closeout chain, while mixed notes require terminal `## Entry Coverage` bullets for every dated entry. When `phase_status` is requested, an existing active-plan frontmatter `status` copy is synchronized to the same value as derived metadata, and the phase block selected by the requested or current `active_phase` gets its body `- status:` line updated. The body status for lifecycle `phase_status = complete` is `done`; other lifecycle status values are copied as-is. `writeback --apply --phase-status complete` is the explicit ready-for-closeout boundary: it keeps the active plan open and does not archive the plan, default roadmap items to `done`, archive source incubation, or open the next slice. `writeback --dry-run --archive-active-plan` and `writeback --apply --archive-active-plan` are the explicit completed-plan close rail for live operating roots using the default active plan and `project/archive/plans/` archive directory: dry-run reports active plan, archive target, lifecycle updates, closeout facts, optional roadmap/source-incubation sync posture, closeout identity carry/replace/refusal posture, archive route-retarget posture, auto-compaction posture, and validation posture; apply moves the plan, sets `plan_status` to `none`, clears `active_plan`, records `last_archived_plan`, refreshes Current Focus and the Memory Routing Roadmap hot pointer, and preserves or updates supplied closeout facts. If any resulting live `project/project-state.md` after writeback or plan state writes exceeds the default 250-line or 25,000-character threshold and section boundaries are clear, auto-compaction archives older history into `project/archive/reference/project-state-history-YYYY-MM-DD.md` or a conflict-free same-day variant and leaves compact operating memory with current focus, memory routing roadmap, repository role map, closeout writeback, and an archive pointer. Selection scans the whole state file: the rail keeps current authority and the latest relevant update, then archives older/non-current history instead of trimming only the newest note that happened to push the file over the threshold. `writeback --dry-run|--apply --compact-only` runs only that state-history compaction rail and refuses combination with closeout/lifecycle/archive or roadmap sync fields. Refused or skipped compaction does not invalidate the state writeback. It does not stage, commit, clean unrelated archives, repair unrelated files, create generated evidence stores, create repair snapshots, change docmaps, or make read-only warnings into approval.

When a successful `writeback --apply` or reviewed `transition --apply` lands lifecycle/archive writes but leaves a single `writeback-incubation-archive-blocked` warning, the Work Result treats the lifecycle work as completed and routes shared or mixed source-incubation coverage to later operating-memory hygiene instead of presenting the closeout as failed.

`writeback --archive-active-plan --phase-status complete` is the same-request close-and-archive form for a pending active plan: it synchronizes completed active-plan copies and archives atomically. Without current or same-request `phase_status = complete`, archive still refuses before writes.

`writeback --apply --active-phase <next-phase> --phase-status pending` is the explicit same-plan phase-advance path: it completes the previously active phase block and advances `project-state` Current Focus to the requested pending phase in one lifecycle writeback. The `writeback-phase-advancement` report remains a boundary finding only and does not approve auto-continue, closeout, archive, roadmap done-status, next-plan opening, staging, or commit.

The implemented mutating active-plan scaffold path is hidden `plan --dry-run|--apply`. Dry-run is no-write. Apply is allowed only for a live operating root with readable frontmatter state and the default manifest plan path, refuses product-source fixtures, fallback/archive roots, ambiguous roots, malformed frontmatter, stale inactive plan files, unsafe plan targets, active-plan conflicts without `--update-active`, destructive task input, and invalid requested roadmap links, and writes only `project/implementation-plan.md` plus selected project-state lifecycle frontmatter. With `--roadmap-item <id>`, it may derive missing title/objective/task input from the roadmap item heading, `slice_goal`, verification context, and source or related incubation note excerpt, then report those candidate inputs with `plan-roadmap-derived-input`, `plan-roadmap-candidate-objective`, and `plan-roadmap-candidate-task`. With `--roadmap-item <id>`, it may also write the requested roadmap item block plus roadmap-slice `covered_roadmap_items` to record `related_plan = project/implementation-plan.md`, refresh the explicit source-incubation note's `related_plan`, `related_roadmap_item`, `related_roadmap`, and `promoted_to` fields, and record `related_roadmap_item`, known source relationships, and executable slice frontmatter in the active plan when the roadmap item exposes it. Related-incubation-only roadmap items are reported as non-owning provenance and do not produce source-note writes. The source-note refresh replaces full targeted relationship scalar blocks so malformed prior list-form values do not leave YAML tail lines behind. Plan-side slice frontmatter includes `primary_roadmap_item`, `covered_roadmap_items`, `domain_context`, `target_artifacts`, `execution_policy`, `auto_continue`, `stop_conditions`, and `closeout_boundary`; dry-run reports that roadmap and slice write separately from lifecycle writes. Generated active plans default to `execution_policy = current-phase-only` and `auto_continue = false`; the safe continuation is the single `active_phase` from `project/project-state.md`, followed by a stop for repo-visible evidence/state. Plan synthesis renders verification gates from target test artifacts or repo-visible command discovery over product/target root evidence such as package scripts, Makefile/just/task files, CI workflow `run:` lines, documented commands, and roadmap verification summaries; weak evidence stays `UNRESOLVED` rather than becoming a universal pytest fallback. When source implementation phases are verified by target test artifacts, plan synthesis reports `plan-adjacent-verification-ownership` and includes those regression tests in the phase write scope. When roadmap metadata already says `docs_decision = updated`, plan synthesis reports `plan-docs-write-scope-impact`, counts that docs update in phase pressure, and names docs/spec/package metadata targets in the generated docs phase; if the roadmap omitted exact docs targets, the phase write scope contains an explicit placeholder that must be replaced before mutation. `transition --dry-run|--apply` can reuse the same derivation for omitted `--next-title`, `--next-objective`, and `--next-task`, while still requiring review-token apply and repo-visible drift checks. `check` reports missing, false, malformed, or unsafe auto-continuation through read-only `active-plan-execution-policy`, `active-plan-auto-continue-*`, and `active-plan-stop-conditions` diagnostics, reports `active-plan-source-incubation-related-plan-drift` when the active plan names a source-incubation note whose `related_plan` no longer points at `project/implementation-plan.md`, and may warn with `active-plan-verification-gate-unresolved`, `active-plan-verification-gate-generic`, `active-plan-verification-gate-toolchain-mismatch`, or `active-plan-adjacent-verification-ownership` when gates are weak, obviously mismatched, or broader than declared regression-test ownership. If a plan explicitly sets `auto_continue = true`, stop conditions must cover verification, authority/docs impact, write scope, source reality, sensitive action, and closeout boundaries before the report can call it a candidate. Lifecycle writeback reports `writeback-phase-execution-boundary` so phase pointer movement is not confused with authorization for auto-continue, closeout, archive, commit, or next-slice movement. The generated plan records `docs_decision = "uncertain"` as derived diagnostic metadata until closeout evidence proves `updated` or `not-needed`; it cannot approve repair, closeout, archive, commit, rollback, or future mutation by itself.

When roadmap slice fields are present, plan synthesis may also report `plan-synthesis-bundle-rationale`, `plan-synthesis-split-boundary`, `plan-synthesis-target-artifact-pressure`, `plan-synthesis-phase-pressure`, `plan-verification-gate-discovery`, `plan-verification-gate-unresolved`, `plan-adjacent-verification-ownership`, and `plan-docs-write-scope-impact`. The generated active plan mirrors sizing signals in a `Plan Synthesis Notes` section with `target_artifact_pressure` and `phase_pressure` values, and mirrors the verification posture in `verification_gates`. These reports explain why items were bundled or split, whether a concrete verification command was discovered, whether docs/spec/package metadata scope is named when docs will be updated, and whether adjacent regression-test ownership is named, but they remain advisory execution evidence only and cannot approve lifecycle movement, closeout, archive, repair, commit, rollback, or future mutation.

The implemented mutating memory lifecycle hygiene path is hidden `memory-hygiene --dry-run|--apply`. Dry-run is no-write. `memory-hygiene --dry-run --scan` is a read-only relationship hygiene scan and incubation cleanup advisor that requires no source path. It classifies active incubation notes as keep-active, archive candidates, entry-coverage-needed, follow-up extraction, exact-link repair candidates, or ambiguous without writing cleanup state. Apply is allowed only for a live operating root with an explicit source under `project/research/` or `project/plan-incubation/`, optional existing `--promoted-to` destination, optional explicit `--archive-to` destination under `project/archive/reference/research/` or `project/archive/reference/incubation/`, and optional exact `--repair-links`. It writes only requested source frontmatter fields such as `status`, `updated`, `promoted_to`, and `archived_to`, terminal Entry Coverage bullets requested with `--entry-coverage`, the explicit or `--archive-covered` derived archive target, and exact root-relative source-path link replacements in MLH-owned lifecycle surfaces. `--archive-covered` is incubation-only and refuses open or malformed Entry Coverage before writing. It refuses product-source fixtures, fallback/archive roots, ambiguous roots, missing or malformed source frontmatter, invalid routes, path escapes, symlinks, target conflicts, scan apply, and `--repair-links` without an archive target; it cannot infer promotion meaning, perform fuzzy link repair, approve archive decisions, closeout, commit, rollback, or broader lifecycle changes.

The implemented mutating roadmap item path is hidden `roadmap --dry-run|--apply`. Dry-run is no-write and reports the target file, action, item id, changed fields, reciprocal source-incubation relationship writes or non-owning reused source-incubation provenance when requested, queue-summary refresh, status-scoped order namespace diagnostics, current-posture refresh when present, and live-tail compaction candidates when older detailed `done` item blocks have durable evidence. Apply is allowed only for a live operating root with readable frontmatter state and an existing regular `project/roadmap.md`. `--action add` requires a canonical `## Items` section with managed `###` item blocks; `--action update`, `plan --roadmap-item`, and `writeback --roadmap-item` may also target existing compatibility top-level `## RM-...` sections. It writes only the target roadmap file plus unowned explicit `source_incubation` note frontmatter when that relationship is requested, appending one new canonical item block for `--action add` or replacing/updating one existing managed item block or RM section for `--action update`; reused source notes are recorded as `related_incubation` on the roadmap item without source-note writes. Explicit source-note writes replace the whole targeted relationship scalar block, including malformed list-block tails left by older metadata. Inside canonical roadmap files, it refreshes the MLH-owned `Future Execution Slice Queue` from accepted/proposed item metadata, treats accepted and proposed order values as separate namespaces, refreshes any existing MLH-owned `Current Posture` section from active/accepted/proposed item metadata, omits empty strict path relationship fields such as `source_incubation`, `related_incubation`, `source_research`, `related_plan`, and `archived_plan` from rewritten canonical item blocks, and may compact older detailed `done` item blocks into `Archived Completed History` when those items already carry archived-plan, verification, and final docs-decision evidence. Roadmap-side slice metadata is advisory and explicit: `execution_slice`, `slice_goal`, `slice_members`, `slice_dependencies`, `slice_closeout_boundary`, `source_research`, `related_specs`, and `target_artifacts` can be recorded as planning evidence without approving lifecycle movement. `plan --roadmap-item` and `writeback --roadmap-item` reuse this managed item validation for explicit relationship sync, with plan opening mapping the active plan path into the requested item plus roadmap-slice `covered_roadmap_items` and refreshing each explicit source-incubation note, and writeback mapping same-request `verification`, `docs_decision`, `residual_risk`, and `carry_forward` into the requested item plus active-plan `covered_roadmap_items`; compatibility RM sections ignore `slice_members` values that are not known roadmap ids. It refuses product-source fixtures, fallback/archive roots, ambiguous roots, malformed state frontmatter, malformed roadmap files, duplicate ids, missing update targets, invalid statuses, invalid docs decisions, unsafe relationship paths, route-incompatible relationship destinations, missing related item ids, missing or malformed source-incubation frontmatter for reciprocal writes, self-relations outside slice membership, symlink targets, and parent path conflicts; it cannot infer roadmap priority, approve repair, approve closeout, archive, commit, rollback, or broader lifecycle changes.

The implemented read-only route metadata diagnostic path is part of `validate`, `check`, and `check --focus validation` for live operating roots. It inspects routed roadmap, research, incubation, stable-spec, decision, ADR, and verification Markdown frontmatter for known lifecycle statuses, malformed relationship values, root-relative path safety, missing targets, archive-destination shape, promotion destinations, route-specific `related_*` fields, `source_research`, `source_incubation`, `source_roadmap`, `implemented_by`, `merged_from`, `merged_into`, `rejected_by`, `split_from`, `split_to`, `archived_plan`, `supersedes`, and `superseded_by`. `project/roadmap.md` is the only accepted roadmap route in this slice, and roadmap relationship fields remain diagnostics only; they do not create or synchronize roadmap files outside the explicit `roadmap --dry-run|--apply` rail. Strict fields such as `promoted_to` and `archived_to` must be root-relative paths; flexible relationship fields are path-validated only when the value is path-like, so historical labels or prose are not treated as one missing local file. Stale-active-reference warnings apply to relationship pointers, while `source_*` provenance may keep pointing at a source artifact that records its own archive. Roadmap order diagnostics report accepted/proposed namespaces separately and warn only for duplicate order values inside the same namespace. It reports warning-level `route-metadata-*` findings plus an authority reminder when problems are found. It does not run for product-source fixtures, does not repair metadata, does not move archives, and does not approve closeout, commit, rollback, or lifecycle decisions; relationship frontmatter block recovery happens only when an explicit roadmap/plan/writeback operation already owns that source-note relationship write.

Doc-gardening checks should report stale pointers, missing required sections, invalid links, path drift, lifecycle conflicts, and forbidden residue before any repair flow mutates files.

Repair snapshot metadata is provenance for one repair attempt. It may record schema version, tool/version, command, root kind, repair class, target paths, copied files, pre-repair hashes, source diagnostics, planned route entries, planned frontmatter keys, retention posture, rollback instructions, and an authority note. It must point back to repo-visible target files and copied bytes, and it must not become a durable evidence database, quality gate, routing authority, closeout approval, archive approval, commit decision, or future repair authorization.

The implemented repair candidates are snapshot-protected state frontmatter repair, create-only `AGENTS.md` creation, create-only `.agents/docmap.yaml` creation, create-only stable workflow spec fixture restoration, and snapshot-protected `.agents/docmap.yaml` route repair. `state-frontmatter-repair` is allowed only when validation reports `state-prose-fallback` for the default manifest state path, required lifecycle assignments are present, and active-plan pointers match the manifest; malformed or partial frontmatter remains manual. `agents-contract-create` is allowed only when validation reports a missing required `AGENTS.md`; existing AGENTS files are preserved, and no snapshot is created because no existing bytes are changed. `docmap-create` is allowed only when validation reports a missing required docmap; lazy or not-required docmaps remain absent, and no snapshot is created because no existing bytes are changed. `stable-spec-create` is allowed only when validation reports missing required `project/specs/workflow/*.md` fixtures; existing stable specs are preserved, and no snapshot is created because no existing bytes are changed. `snapshot --inspect` reads repair snapshot metadata and copied bytes as safety evidence, reporting schema, repair class, command, target-root and snapshot-root posture, copied-file presence, hash and byte-count consistency, current-target posture, planned frontmatter keys when present, retention, manual rollback instructions, and non-authority wording without approving repair or rollback. `repair --dry-run` reports target files, deterministic preview snapshot paths where a snapshot class applies, copied-file paths, source diagnostics, planned frontmatter keys, route entries, AGENTS contracts, or stable spec files, metadata fields, manual rollback posture, and `validate`/`audit-links` validation methods. `repair --apply` writes timestamped snapshot metadata and copied pre-repair bytes before prepending state frontmatter or adding missing route entries to an existing docmap, and uses packaged templates for create-only AGENTS and stable spec restoration. Snapshot metadata may help an operator inspect a bounded rewrite, but it cannot make operator-contract text, docmap routing, state metadata, or stable spec fixtures authoritative, and it cannot promote manifest/archive/active-plan repair.

## Future Product Gates

Before metadata, routing, Git-evidence helpers, quality gates, or stronger evidence helpers are implemented, a later scoped plan must define:

- minimal metadata fields and tolerant defaults
- required versus optional lifecycle fields
- severity levels for missing, stale, or contradictory metadata
- routing-map ownership and update rules
- stronger Git-evidence behavior beyond the current read-only trailer suggestions, including any field-shape changes, generated output, enforcement, or non-git fallback changes
- persistent generated evidence manifest shape, identity, and provenance fields only if a later plan explicitly re-accepts generated manifests after Git-native evidence proves insufficient
- repair snapshot metadata, retention, and cleanup rules when existing-content repair is in scope
- quality-gate names, skip rules, and closeout linkage when enforcement or durable gate state is in scope
- stale-link and stale-pointer report format
- migration behavior for existing docs
- tests for metadata absence, partial validity, stale routing, and conflicting lifecycle status

Strict validators must not block recovery when a human-readable file still contains enough authority for read-only work.

## Validation Expectations

A valid implementation should prove that:

- documents remain understandable without metadata
- metadata routes to existing files
- stale metadata is reported rather than silently trusted
- evidence points to inspectable artifacts or explicit skip rationale
- terminal evidence reports remain useful without storing generated truth
- report-only evidence cue identity remains source-bound and deterministic without becoming a manifest
- durable proof/evidence records under `project/verification/*.md` are surfaced as optional report inputs without satisfying closeout fields by themselves
- manifest or prospective closeout wording remains context rather than concrete closeout completion evidence
- closeout assembly reports preserve Git/VCS facts and any suggested trailers as advisory evidence rather than required authority
- report-only quality gates improve closeout readiness without enforcement state
- generated reports can be deleted and rebuilt when a later generated-report feature exists
- routing never authorizes mutation by itself
- repair snapshot metadata and `snapshot --inspect` remain evidence for copied bytes, not authority for repair, rollback, cleanup, or closeout
- writeback apply synchronizes closeout/state authority, the marked Current Focus block, and derived active-plan copies in the same mutation, while read-only commands keep reporting stale derived copies without writing
- archive-active-plan writeback dry-run/apply closes the default completed-plan lifecycle without turning repair, check, evidence, or closeout reports into archive approval
- post-writeback, post-plan, and compact-only auto-compaction keep oversized live operating memory compact without giving read-only context warnings permission to write files
- roadmap dry-run/apply can add or update one explicit managed item block, compact eligible completed queue/live-tail history into archived references, and preserve roadmap authority boundaries
- plan/writeback roadmap sync can update an explicit roadmap relationship, and writeback archive closeout can batch active-plan covered roadmap items, without making roadmap output lifecycle approval
- plan --roadmap-item can derive active-plan slice frontmatter from repo-visible roadmap fields without putting slice membership into project-state
- live-root status/check summarize completed active phases as explicit closeout/archive/manual-commit next action rather than leaving agents to reconstruct that state from prose history
- legacy reference material is opened only for a named blocker

Validation should prefer report-first checks. Mutating doc repair requires a separate explicit plan.

## Explicit Non-Goals

- No metadata schema implementation in this phase.
- No evidence database, hidden registry, default persistent evidence manifest, or generated truth.
- No Git staging, commits, amendments, pushes, hook installation, or Git config mutation by evidence helpers.
- No required docmap for a first operating-root phase.
- No auto-gardening or silent doc repair.
- No metadata-only authority.
- No quality-gate ceremony for every small ad hoc task.
- No import of old research or archives as routing defaults.
