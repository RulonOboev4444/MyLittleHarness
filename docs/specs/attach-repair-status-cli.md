# Attach Repair Status CLI Spec

## Purpose

This spec defines the product boundary for MyLittleHarness CLI surfaces that attach, check, repair, detach, and expose advanced diagnostics for repository contracts.

The CLI contract must be explicit before product code implements commands that inspect or mutate operating memory, fixtures, or reusable product docs.

## Authority

Repo-visible files remain authoritative. A CLI may discover, summarize, validate, or propose changes, but accepted workflow truth remains in the target repository operating root and stable product contracts.

For the portable clean-room posture:

- the operating project root is the repository MyLittleHarness is servicing and owns active operating memory and plans for that repository
- the product source checkout owns reusable product source and docs
- legacy reference material is opened only for a named blocker
- MyLittleHarness serves the explicit target repository directly

The CLI must preserve the distinction between live operating memory and product-root compatibility fixtures.

## Non-Authority

CLI output is not authority by itself. Diagnostics, generated reports, dry-run plans, stdout, exit codes, caches, and SQLite rows must not hold the only copy of accepted decisions, active focus, plan status, or stable workflow rules.

A successful status or validate run does not approve lifecycle decision, close a plan, archive history, or authorize repair. A repair proposal does not become accepted until a later mutating path writes repo-visible files under an explicit plan.

## Target Visible CLI

The target product CLI is a small repo utility:

- `init`: attach MyLittleHarness to an explicit target repository. Current implementation routes `init --dry-run` and `init --apply --project <name>` through the same bounded attach behavior as the compatibility `attach` command.
- `check`: report orientation, validation, compact check-level drift, and live-root lifecycle routes without writing. Current implementation composes `status`, `validate`, and a small docmap/root-pointer drift section; deeper diagnostics remain explicit advanced commands.
- `repair`: preview or apply one bounded repair class at a time. Current `repair --dry-run` and `repair --apply` remain the product safety model.
- `detach`: preview harness disconnect posture and apply a marker-only disabled posture with no surprise deletion. Current `detach --dry-run` is read-only and preservation-first; `detach --apply` creates only the disabled marker in eligible live operating roots.

First-contact docs and top-level help show this small shape first. The first-run operator path starts from source-checkout usage with `PYTHONPATH=src`, optionally verifies package metadata and the console script with `bootstrap --package-smoke`, then points `--root` at the target repository for `init --dry-run`, `check`, `repair --dry-run`, and `detach --dry-run`. Advanced diagnostics may remain available through compatibility commands, hidden/deemphasized help, or an explicit advanced namespace, but they should not define the product front door.

The no-skill start pass is part of the CLI contract. A file-reading, shell-capable agent reads `AGENTS.md`, `.codex/project-workflow.toml`, `project/project-state.md`, and `project/implementation-plan.md` only when `plan_status = "active"` or the operator asks about plan/phase/closeout, then runs `check` before mutation. `.agents/docmap.yaml`, `audit-links`, `check`, and relevant specs provide portable docs-decision inputs; no `docs-impact`, `guide`, or `orient` command is required for v1.

`sync` is not a primary command. Disposable projection refresh, search, semantic readiness, adapter projection, and report assembly stay advanced unless a later implementation proves a visible refresh command reduces operator ceremony.

Compatibility is a transition policy:

- Existing commands and exit semantics must remain intact during the first CLI simplification slice.
- Old commands may become aliases, advanced commands, or compatibility surfaces after primary `init` and `check` exist.
- Deprecation starts in docs/help. Normal command stdout should not gain churny warnings unless a later versioned deprecation slice explicitly accepts that behavior.
- Command removal requires a later scoped deprecation/removal slice with tests and migration notes.

## Current Contract

The implemented 1.0.0 product contract is:

- `init --dry-run` and `init --apply --project <name>` are the primary spellings for bounded scaffold attachment. They preserve the existing attach dry-run/apply output shape, mutation boundary, and exit behavior. Valid already-attached live roots report `attach-already-attached` as a clean no-op instead of first-run template conflicts, and already-attached apply skips generated projection writes. First-time successful attach still builds disposable generated projection artifacts and the SQLite index.
- `check` is read-only. Bare `check` is sectioned as `Status`, `Validation`, `Drift`, and `Boundary`. It composes `status`, `validate`, and check-level drift findings for candidate docmap gaps, stale configured-root role wording, primary instruction-surface size warnings, and explicit delivered-vs-remainder capability token contradictions in operating-memory or research surfaces, writes no files, returns `1` when included findings have error severity, and otherwise returns `0` even when warning findings are present. For live operating roots, `Status` includes the compact lifecycle route table, including optional `project/roadmap.md` sequencing when planning context needs it; product-source fixtures do not emit live route-table rows. The primary instruction-surface size warning is limited to start-pass surfaces such as `AGENTS.md`, `README.md`, `.agents/docmap.yaml`, `.codex/project-workflow.toml`, `project/project-state.md`, and the active plan; product docs and stable specs remain covered by advanced `context-budget` detail instead of check-level drift, and their large measurements are informational unless the same surface is also a primary instruction surface. Oversized live `project/project-state.md` warnings may point operators at `writeback --dry-run --compact-only`, but `check` itself remains no-write.
- `check --deep` is read-only and renders `Status`, `Validation`, `Drift`, `Links`, `Context`, `Hygiene`, `Grain`, and `Boundary`. It consolidates the common read-only compatibility diagnostics behind `check` while preserving the old command spellings.
- `check --focus validation|links|context|hygiene|grain` is read-only and renders exactly one focused diagnostic section plus `Boundary`. The focus values route to `validate`, `audit-links`, `context-budget`, `doctor`, and report-only grain diagnostics respectively, with old wrappers preserved for compatibility.
- Grain diagnostics inspect active-plan frontmatter, covered roadmap items, target artifacts, phase count, write-scope and verification specificity, unsafe auto-continue posture, roadmap done/evidence hygiene, targetless non-closed roadmap items, and recent archived-plan calibration samples. These findings are sizing and hygiene signals only: they do not write files, infer roadmap status, approve repair, closeout, archive, commit, rollback, lifecycle decisions, or promote numeric thresholds into product gates.
- `roadmap --dry-run --action add --item-id <id> --title <title> --status <status> --order <int>` and `roadmap --dry-run --action update --item-id <id> [fields]` report the target roadmap file, action, item id, changed fields, reciprocal source-incubation relationship writes, refusal posture, and write boundary without writing files. `roadmap --apply` writes only `project/roadmap.md` plus the explicit `source_incubation` note frontmatter in a live operating root by appending one managed item block or replacing one existing managed item block. It accepts advisory slice fields `--execution-slice`, `--slice-goal`, repeated `--slice-member`, repeated `--slice-dependency`, `--slice-closeout-boundary`, `--source-research`, repeated `--related-spec`, and repeated `--target-artifact` without making roadmap grouping lifecycle authority. It requires a readable `project/project-state.md` frontmatter and an existing regular roadmap file with `## Items`; refuses product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, malformed roadmaps, duplicate ids, missing update targets, invalid statuses, invalid docs decisions, unsafe relationship paths, incompatible relationship destinations, missing related item ids, missing or malformed source-incubation frontmatter for reciprocal writes, self-relations outside slice membership, and symlink/path conflicts; and it does not infer priority, repair files, archive, stage, commit, approve closeout, or make roadmap output lifecycle approval.
- `detach --dry-run` is the primary detach preview. It is explicit-mode only, returns `0` for readable roots, renders `Root Posture`, `Preservation`, `Marker`, `Generated Projection`, `Manual Recovery`, and `Boundary` sections, uses `detach-*` finding codes, previews `.mylittleharness/detach/disabled.json`, and writes no files, reports, caches, generated outputs, snapshots, Git state, config, hooks, CI files, package artifacts, or workstation state. Bare `detach`, combined modes, unknown flags, and `disable` are argparse usage failures with exit code `2`.
- `detach --apply` is the marker-only detach apply path. It creates only `.mylittleharness/detach/disabled.json` in eligible live operating roots, leaves valid existing markers for the same root unchanged, returns `0` for marker creation or idempotent reuse, and returns `2` when target root authority, default authority paths, marker path, or existing marker payload is invalid or ambiguous. Product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, missing or unreadable manifest/state surfaces, non-default authority paths, symlink/path conflicts, and boundary conflicts are refused before marker creation. `disable` is explanatory terminology only, not a CLI alias.
- `detach` preserves `.codex/project-workflow.toml`, `project/project-state.md`, `.agents/docmap.yaml`, `project/specs/workflow/`, archives, research, repair snapshots, and `.mylittleharness/generated/projection/` when present. The disabled marker is informational evidence only and cannot approve cleanup, repair, closeout, archive, commit, rollback, lifecycle decisions, or future mutations.
- `status` is read-only and reports current root posture, manifest presence, state/plan pointers, fixture classification, obvious residue risks, a live-root lifecycle continuation summary, and the live-root lifecycle route table when the target is a live operating root.
- `validate` is read-only and checks contract structure, required files, path validity, lifecycle status, and cross-link sanity.
- Read-only status and validation distinguish product-source fixtures from live operating roots. For live operating roots, a root `README.md` is optional, `.agents/docmap.yaml` may remain absent when the manifest uses `docmap_mode = "lazy"`, and prose-style `project/project-state.md` assignment lines may supply read-only lifecycle fields such as `operating_mode`, `plan_status`, `active_plan`, `active_phase`, and `phase_status`.
- `context-budget`, `audit-links`, and `doctor` are read-only analyzer commands that report context size, local link/docmap/root-pointer drift, and product-root hygiene. Product docs and stable specs stay visible in context-budget measurements, but size alone does not turn them into warning pressure. `doctor` stays summary-oriented and reports context-budget warning counts instead of duplicating section-size output. `audit-links` treats repair snapshot copied-file paths such as `files/.agents/docmap.yaml` as snapshot-internal references when they are absent from the repo root.
- `intelligence [--query TEXT] [--search TEXT] [--path TEXT] [--full-text TEXT] [--limit N] [--focus search|warnings|projection|routes]` is an advisory aggregate command. Its default output reports terminal-only sections named `Summary`, `Boundary`, `Drift`, `Repo Map`, `Backlinks`, `Search`, `Fan-In`, and `Projection`. Exact-only and route/projection views stay no-write; path/reference and full-text navigation may refresh disposable generated projection cache under `.mylittleharness/generated/projection/` when missing or stale.
- `intelligence` uses only inventory-discovered surfaces as its initial search and backlink corpus. Exact text search and path/reference search are case-sensitive, `--path` remains a path/reference query rather than a report scope, and `--query` expands a single recovery query into any omitted exact text, path/reference, and full-text modes while explicit mode flags keep their own values.
- `intelligence --focus search` renders compact source inventory plus `Summary` and `Search` only. Exact text search reads direct source content through the in-memory projection because projection artifacts do not store source bodies. Path-focused search compares valid artifact path/reference rows with the current in-memory projection after refreshing missing or stale artifacts when the boundary is safe; degraded artifacts still fall back to direct in-memory path search. Full-text search refreshes a missing or stale SQLite FTS/BM25 index when the boundary is safe, then uses only a current source-verified index; if refresh is refused, corrupt, malformed, or FTS5-unavailable, it reports an advisory skip reason. Plain multi-term full-text input is relaxed into an OR query over indexable terms for recovery search, while explicit uppercase FTS operators such as `AND`, `OR`, `NOT`, or `NEAR`, quoted input, and other FTS control markers keep explicit query mode. `--limit` controls full-text result count and must be at least `1`. `intelligence --focus warnings` renders compact source inventory plus `Summary` and actionable warning/error findings only, including primary instruction-surface size warnings and remainder-drift warnings when the same explicit capability token is described as delivered/current and future/backlog work in operating-memory or research surfaces outside historical-release context.
- `intelligence --focus projection` renders compact source inventory plus `Summary`, `Boundary`, and `Projection` only. Projection output reports rebuild status, in-memory storage boundary `none`, owned artifact boundary status, source-hash coverage, link/fan-in/relationship record counts, missing/unreadable source diagnostics, and source provenance.
- `intelligence --focus routes` renders compact source inventory plus `Boundary` and `Lifecycle Routes` only. For live operating roots, it reports the same advisory route table as `status`/`check`: `state`, `active-plan`, `incubation`, `research`, `stable-specs`, `decisions`, `adrs`, `verification` for the active-plan block plus optional `project/verification/*.md` proof/evidence records, `closeout-writeback`, `archive`, and `docs-routing`. For product-source fixtures, fallback/archive roots, generated-output roots, or ambiguous roots, it reports scope instead of live route rows.
- Focused and default `intelligence` output may demote optional/lazy/user-global/historical recovery noise to informational findings before computing the intelligence result. This demotion is scoped to `intelligence` and does not change `audit-links`, `doctor`, or validation severity semantics.
- `intelligence` rebuilds an in-memory projection from `Inventory` on every run and may derive repo-map rows, backlink references, search matches, fan-in summaries, relationship graph records, source hashes, and record counts from that projection. It may write or refresh only disposable projection artifacts and SQLite index files inside `.mylittleharness/generated/projection/` when path/reference or full-text navigation needs them. It must not write report files, generated output outside that boundary, adapter state, hooks, mutation proposals, workflow execution surfaces, repair snapshots, archives, lifecycle state, or authority files.
- `evidence` is a read-only terminal report with no mode flags. It scans inventory-discovered workflow surfaces for active-plan presence, source-set pointers, verification anchor candidate text, optional durable proof/evidence records under `project/verification/*.md`, closeout field candidates, validation sections as verification closeout candidates, residual-risk lines, explicit skip rationale, and carry-forward destination mentions. Candidate cue findings include report-only identity metadata from cue kind, source path, line number, normalized preview text, and a deterministic hash. Concrete closeout field candidates require explicit field bullets, exact field headings, or observed result lines; manifest language and durable proof/evidence records remain context until explicit closeout fields say how they were used. It reports `worktree_start_state` and `task_scope` as operator-required reminders instead of running Git or any VCS subprocess, and it remains read-only.
- `evidence` finding codes use the `evidence-*` family. Findings may report `candidate`, `missing`, `ambiguous`, `operator-required`, identity, or quality-readiness posture, but they must not report helper decisions such as complete, approved, closed, committed, or archived. Missing active plans, absent closeout fields, prose-only state, non-git roots, and product-source fixtures remain report states rather than command failures.
- `closeout` is a read-only terminal report for closeout assembly. It may run only target-bound read-only VCS probes equivalent to `git -C <root> rev-parse --is-inside-work-tree`, `git -C <root> rev-parse --show-toplevel`, and `git -C <root> status --porcelain=v1`. It must not run Git write operations, remote/network operations, hook/config mutation, stash, checkout, reset, clean, fetch, pull, push, worktree creation, archive, repair, commit, or lifecycle mutation.
- `closeout` finding codes use the `closeout-*` family, with Git evidence suggestions under `closeout-git-evidence-*`. The report assembles candidates and prompts for `worktree_start_state`, `task_scope`, `docs_decision`, `state_writeback`, `verification`, and `commit_decision`, plus optional durable proof/evidence record prompts, residual-risk, explicit-skip, carry-forward, report-only quality/readiness cues, projection posture, manifest policy, root-kind cues, and optional read-only Git trailer suggestions. Concrete closeout field lines are explicit field bullets, exact field headings, observed result lines, or fields in the MLH closeout writeback block. Broad future-contract mentions and durable proof/evidence records remain context when concrete field evidence is absent. It may report clean, dirty, non-git, Git-missing, and Git-failure posture, including bounded changed-file counts and samples from porcelain status. For Git worktrees, trailer suggestions prefer the current project-state writeback block over stale derived active-plan copies and are emitted only when explicit closeout facts provide the required fields; for non-git or unknown Git posture, the report gives the Markdown/operator-summary fallback and emits no trailer suggestions. It does not approve closeout, archive plans, decide task scope, stage files, commit, write state, write persistent evidence manifests, or write quality-gate state.
- `docs_decision` values are `updated`, `not-needed`, and `uncertain`. A docs decision is required when behavior, CLI usage, configuration, setup, contract meaning, permissions, output shape, UX/copy, terminology, rollout, migration, or other user-facing meaning changes. `uncertain` blocks confident closeout language. The decision is an operating-root closeout fact assembled from diff evidence, docmap routing when present, `audit-links`, `check`, and relevant specs; it is not owned by Codex skills, IDE rules, generated reports, MCP clients, hooks, or CI. The current MLH-owned closeout authority is the marked closeout writeback block in `project/project-state.md`; active-plan frontmatter `docs_decision` is diagnostic derived metadata. `validate` treats active-plan frontmatter `docs_decision` as diagnostic metadata: unsupported values are warnings, and `docs_decision = uncertain` is warned when state frontmatter already records `phase_status = complete`.
- `plan --dry-run --title <title> --objective <objective>` reports the deterministic active-plan scaffold target, lifecycle frontmatter update, docs-decision posture, root posture, refusal posture, execution-policy posture, and write boundary without writing files. `plan --apply --title <title> --objective <objective>` writes only `project/implementation-plan.md`, selected lifecycle frontmatter in `project/project-state.md`, and the marked Current Focus block derived from that frontmatter, with optional `--task <text>` and `--update-active` fields: `operating_mode`, `plan_status`, `active_plan`, `active_phase`, and `phase_status`. With `--roadmap-item <id>`, dry-run separately reports the roadmap relationship and slice-frontmatter update; apply writes only the requested roadmap item block plus roadmap-slice `covered_roadmap_items` to set `related_plan = project/implementation-plan.md` and derives active-plan frontmatter such as `primary_roadmap_item`, `covered_roadmap_items`, `domain_context`, `target_artifacts`, `execution_policy`, `auto_continue`, `stop_conditions`, and `closeout_boundary` from repo-visible roadmap fields. Missing or malformed roadmap items refuse before the plan is created. It renders a compact repo-visible plan with phase, execution-policy, file ownership, verification, docs-decision, state-transfer, refusal, and closeout sections; starts `docs_decision` as `uncertain`; defaults to current-phase-only execution with `auto_continue = false`; records stop conditions for verification, authority/docs impact, write scope, source reality, sensitive action, and explicit closeout boundaries; refuses product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, malformed or missing state frontmatter, non-default plan paths, unsafe path/symlink targets, stale inactive plan files, existing active-plan conflicts unless `--update-active` is supplied, and task input that asks for destructive VCS recovery or broad cleanup. It does not infer architecture from chat, call an LLM, install dependencies, stage, commit, archive, repair unrelated files, or approve lifecycle closeout.

- With `--roadmap-item <id>`, `plan --dry-run` and `plan --apply` also report bounded synthesis findings when roadmap slice metadata is present: `plan-synthesis-bundle-rationale`, `plan-synthesis-split-boundary`, `plan-synthesis-target-artifact-pressure`, and `plan-synthesis-phase-pressure`. The generated plan may include `Plan Synthesis Notes` with `target_artifact_pressure` and `phase_pressure`. These findings are advisory sizing evidence only; they do not create hard thresholds or approval for lifecycle movement, closeout, archive, repair, commit, rollback, or future mutations.
- `memory-hygiene --dry-run --scan` reports relationship hygiene, text-input audit posture, Entry Coverage completeness, and heuristic split suggestions without writing files or requiring a source path. `memory-hygiene --dry-run --source <rel> [--promoted-to <rel>] [--status <value>] [--archive-to <rel>] [--repair-links]` reports the source, lifecycle frontmatter update, explicit archive target, optional exact-link repair count, root posture, refusal posture, and write boundary without writing files. `memory-hygiene --apply` writes only the declared MLH-owned research/incubation source frontmatter, explicit archive target, and optional exact source-path link repairs in a live operating root. It refuses product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, invalid source routes, missing or malformed source frontmatter, missing promoted targets, unsafe path/symlink targets, archive target conflicts, scan apply, and `--repair-links` without `--archive-to`. It does not infer promotion meaning, perform fuzzy link repair, stage, commit, approve closeout, or make archive movement an authority signal.
- `writeback --dry-run` reports the planned closeout/state writeback without writing files. `writeback --apply` writes the MLH closeout writeback block in `project/project-state.md`, requested `active_phase`, `phase_status`, or `last_archived_plan` lifecycle frontmatter in `project/project-state.md`, the marked Current Focus block derived from lifecycle frontmatter, and matching derived active-plan frontmatter/body fields when the active plan exists and those closeout copies are in frontmatter or explicit closeout summary/facts/fields sections. Lifecycle-only phase-pointer writeback does not carry previous closeout facts into the active plan. It synchronizes one-line `worktree_start_state`, `task_scope`, `docs_decision`, `state_writeback`, `verification`, `commit_decision`, `residual_risk`, and `carry_forward` values in one apply, refusing multiline closeout field input before writes; when `phase_status` is written, an existing active-plan frontmatter `status` copy is synchronized to the same value and the phase block selected by `active_phase` has its body `- status:` line updated. Lifecycle `phase_status = complete` maps to body status `done`; other status values are copied as-is. With `--roadmap-item <id>`, writeback separately previews or applies one roadmap item update using only selected same-request closeout facts: `verification` becomes `verification_summary`, `docs_decision` becomes `docs_decision`, residual risk and carry-forward are folded into the roadmap `carry_forward` field, and the active or archived plan path is written to `related_plan` and `archived_plan` when available. When that roadmap item has `source_incubation`, archive close may also sync same-request closeout facts into that note and auto-archive it to `project/archive/reference/incubation/**` only when it is structurally covered; single-entry notes can be covered by the closeout chain, while mixed notes require terminal `## Entry Coverage` bullets for every dated entry and otherwise stay active with blocker findings. `--roadmap-status <status>` sets an explicit roadmap status; archive close defaults the roadmap status to `done` when no roadmap status is supplied. `writeback --dry-run --archive-active-plan` reports the active plan, planned archive target, planned lifecycle updates, closeout facts, auto-compaction posture, roadmap/source-incubation sync posture when requested, and validation posture without writing. `writeback --apply --archive-active-plan` is live-root only, moves the default `project/implementation-plan.md` to `project/archive/plans/<date-and-safe-slug>.md`, sets `plan_status = "none"`, clears `active_plan`, records `last_archived_plan`, refreshes Current Focus, and preserves or updates supplied closeout writeback facts. `writeback --apply --phase-status complete` is the explicit ready-for-closeout boundary: it synchronizes state and active-plan derived copies, keeps the active plan open, and does not archive the plan, default roadmap items to `done`, archive source incubation, or open the next slice. Unsafe explicit archive or incubation auto-archive targets refuse before writes, and predictable roadmap/source-incubation sync errors refuse before lifecycle writes. After any explicit live-root `writeback --apply` or safe `plan --apply` state write, if `project/project-state.md` is above the default 250-line threshold, state auto-compaction may archive older dated/history sections to `project/archive/reference/project-state-history-YYYY-MM-DD.md` or a conflict-free same-day variant and leave compact operating memory with frontmatter, title, current focus, memory routing roadmap, repository role map, short notes, the latest relevant update, the current MLH closeout writeback block, and an `Archived State History` pointer. `writeback --dry-run --compact-only` and `writeback --apply --compact-only` run only that compaction rail and refuse combination with closeout fields, lifecycle fields, archive, or roadmap sync fields. Dry-run reports whether compaction would run, skip, or refuse, the target archive path, sections that would stay, sections that would be archived, and validation posture. Compaction refuses or skips product-source fixtures, fallback/archive/generated roots, ambiguous roots, missing or malformed state frontmatter, unsafe or non-default state paths, non-regular or symlink state files, unclear section boundaries, path escapes, and unresolved archive target conflicts; compaction refusal does not undo a safe state writeback. It does not repair docmaps, repair snapshots, generated projection output, clean unrelated archives, stage, commit, mutate Git config, hooks, adapters, or workstation state.

## Mutation Boundary

Mutating attach and repair behavior is gated by root classification before any filesystem write:

- A live operating root is an operator-provided target repository that owns operating memory and is not classified as product source, product compatibility fixture, legacy reference material, generated output, adapter state, cache, or archive-only material.
- A product-source compatibility fixture is any target whose state marks `root_role = "product-source"`, `fixture_status = "product-compatibility-fixture"`, or whose `product_source_root` resolves to the target root. Apply modes must refuse this target with exit code `2`; dry-run modes remain report-only and may return no-op proposals.
- A fallback, archive, generated-output, adapter, cache, log, local database, package archive, user config, PATH, hook, MCP, GitHub, browser, IDE, or workflow execution surface is outside the write boundary. Apply modes must refuse it with exit code `2`.

All allowed write paths must normalize under the explicit target root. A write that would escape the target root, follow ambiguous authority, rewrite an existing file, delete content, move content, archive content, commit changes, mutate user/workstation configuration, or infer authority from stale docs, weak semantic matches, generated projections, caches, adapters, or terminal output is forbidden.

The `projection` command is the manual refresh/recovery surface for generated artifacts and indexes after the initial attach-time build. It may write and delete generated artifacts and the SQLite index only under `.mylittleharness/generated/projection/`; those generated outputs cannot authorize attach, repair, archive, commit, plan lifecycle changes.

Read-only root classification is broader than mutation authority. Prose-style project-state assignments can make `status` and `validate` understand an operating root. The only mutating path from prose state is the selected `state-frontmatter-repair` class, which snapshots default-path `project/project-state.md`, prepends deterministic frontmatter, runs validation, and then stops before any other repair class.

Closeout/state writeback is a separate mutation boundary from repair. `writeback --apply` requires strict readable `project/project-state.md` frontmatter because it writes current closeout facts, lifecycle frontmatter, and the derived Current Focus block rather than recovering missing state structure. It must update all MLH-owned derived active-plan copies it can safely identify during closeout synchronization, so stale `docs_decision: uncertain`, active-plan `status: active`, an active-phase body `- status: pending`, or a stale managed Current Focus block after a completed phase is a diagnostic state, not the expected result of a successful closeout writeback. Schema examples and closeout checklist items are not derived copies. With `--from-active-plan`, exact active-plan closeout bullets are harvested as same-request closeout facts before preflight, and explicit CLI closeout fields take precedence over harvested values. Closeout writeback records plan identity fields when available and reports whether prior facts are carried, replaced, or refused; partial closeout updates are refused when the recorded identity is absent or different from the current active or archived plan unless the same request supplies complete replacement facts. `writeback --apply --phase-status complete` writes the ready-for-closeout boundary without archiving, defaulting roadmap status, source-incubation archival, or next-slice movement. `writeback --apply --archive-active-plan` closes only the default active-plan lifecycle by moving the active plan into `project/archive/plans/` and clearing active pointers while preserving supplied or harvested facts. Optional `--roadmap-item` sync is target-bound to the requested roadmap item plus active-plan `covered_roadmap_items` and uses same-request facts rather than carried project-state closeout values. Archive closeout also retargets MLH route frontmatter that explicitly pointed at the active plan to the new archived plan path. `transition --apply` can compose phase completion, active-plan archive, and next-plan opening only after a matching dry-run review token, and token drift refuses before writes. Reruns after an inactive archive close are refused/skipped without corrupting files. State auto-compaction runs after explicit live-root writeback and plan state writes that leave the state file above the threshold, and `writeback --apply --compact-only` can run the same bounded compaction without other state changes. A compaction refusal or skip is reported as hygiene posture rather than a rollback instruction for the state writeback.

Incubation writeback is also separate from repair. `incubate --apply` requires strict readable `project/project-state.md` frontmatter because it writes a new operating-memory note rather than recovering missing state structure. It owns only `project/plan-incubation/<safe-topic-slug>.md`, refuses product fixtures and unsafe path posture before writing, and leaves promotion into research, specs, plans, or state as a separate accepted-workflow decision.

Memory lifecycle hygiene is also separate from repair. `memory-hygiene --apply` requires an explicit source under `project/research/` or `project/plan-incubation/`, optional accepted `--promoted-to` destination, optional explicit `--archive-to` destination under `project/archive/reference/research/` or `project/archive/reference/incubation/`, and optional `--repair-links` for exact root-relative source-path replacement only. It refuses product fixtures, fallback/archive/generated roots, ambiguous roots, missing or malformed source frontmatter, invalid source routes, path escapes, symlinks, non-directory parent segments, and archive destination conflicts before writing.

Roadmap item mutation is also separate from repair. `roadmap --apply` requires an existing `project/roadmap.md` with a `## Items` section and managed item blocks, plus explicit add/update fields. Plan/writeback roadmap sync uses the same managed item shape and refuses missing or malformed roadmap targets before selected lifecycle writes when the sync is requested. It refuses product fixtures, fallback/archive/generated roots, ambiguous roots, malformed state frontmatter, malformed roadmap files, duplicate ids, invalid item ids, invalid statuses, invalid docs decisions, unsafe relationship paths, incompatible relationship destinations, missing related item ids, self-relations, symlinks, non-directory parent segments, and target conflicts before writing only the roadmap file.

Route metadata diagnostics are read-only validation, not repair. `validate`, `check`, and `check --focus validation` may emit `route-metadata-frontmatter`, `route-metadata-status`, `route-metadata-field`, `route-metadata-path`, `route-metadata-destination`, `route-metadata-missing-target`, `route-metadata-stale-reference`, and `route-metadata-authority` for live operating roots. Those findings may guide manual review of `status`, strict path fields such as `promoted_to` and `archived_to`, and path-like values in flexible relationship fields such as `related_*`, `source_research`, `source_incubation`, `supersedes`, and `superseded_by`. Non-path labels in flexible relationship fields are treated as prose, and stale-active-reference warnings apply to relationship pointers rather than source provenance. Snapshot storage is not a generated projection boundary, adapter surface, cache, or authority layer. ## Repair Snapshot Contract

The MyLittleHarness repair subsystem owns the repair snapshot contract, but the operating root owns the snapshot files. A snapshot is safety evidence for a specific repair attempt, not workflow memory and not accepted truth.

Snapshot storage uses one target-bound boundary:

- Boundary: `.mylittleharness/snapshots/repair/` under the explicit target root.
- Snapshot directory name: `<YYYYMMDDTHHMMSSZ>-<repair-class>-<target-slug>-<hash-prefix>`.
- Required files: `snapshot.json` metadata plus byte-for-byte copies of pre-repair target files under `files/`.
- Path rule: every metadata path and copied file path must normalize under the explicit target root; symlink path segments, non-directory boundary conflicts, path traversal, absolute target paths inside metadata, and existing non-directory snapshot parents are refusal reasons.
- Product hygiene: snapshots must never be created in a product-source checkout or compatibility fixture. Product-root snapshots are debris, not fixtures.

Snapshot metadata must be inspectable and source-bound. The minimum metadata shape is:

- `schema_version`
- `created_at_utc`
- `tool_name` and `tool_version`
- `command`
- `root_kind`
- `repair_class`
- `target_root`
- `snapshot_root`
- `target_paths`
- `copied_files`
- `pre_repair_hashes`
- `planned_post_repair_paths`
- `source_diagnostics`
- `planned_route_entries`
- `retention`
- `rollback_instructions`
- `authority_note`

The `authority_note` must state that snapshots cannot approve repair, closeout, archive, commit, lifecycle decisions, or future repairs. A successful snapshot proves only that pre-repair bytes were copied and metadata was written before mutation started.

Retention is manual-first. MyLittleHarness must not silently delete, rotate, compress, move, or hide repair snapshots. Cleanup is an operator action after review/closeout, or a later explicit cleanup command with its own dry-run, refusal rules, and docs. Reports may warn about old or numerous snapshots, but warnings do not authorize deletion.

The implemented snapshot repair classes are `project/project-state.md` state frontmatter repair and `.agents/docmap.yaml` route repair:

- The state frontmatter target path is limited to `project/project-state.md`.
- The state frontmatter dry-run plan reports deterministic frontmatter keys from existing assignment fallback data, preview snapshot path `.mylittleharness/snapshots/repair/00000000T000000Z-state-frontmatter-repair-project-project-state-md-<hash-prefix>`, metadata fields including `planned_frontmatter_keys`, copied-file path `files/project/project-state.md`, manual rollback posture, and validation method.
- The state frontmatter apply path uses a real UTC timestamp, writes the snapshot before prepending frontmatter, preserves existing body bytes, runs validation, and stops before scaffold/docmap classes.
- State frontmatter repair cannot make lifecycle state authoritative by itself and cannot repair manifest paths, malformed frontmatter, active plans, archives, stable specs, research files, generated outputs, or root-boundary posture.

## Rollback And Idempotency

Create-only scaffold, AGENTS creation, docmap creation, and stable spec creation operations do not require backups because they never overwrite existing content. The implemented state frontmatter repair and docmap route repair require snapshots before rewriting existing files.

Any operation that would overwrite, delete, rename, normalize, resync, archive, or otherwise change existing content outside selected `project/project-state.md` frontmatter prepending or `.agents/docmap.yaml` route-entry update is blocked unless a later implementation uses the repair snapshot contract above and starts with a no-write dry-run snapshot plan.

Rollback is manual-only for this contract. A create-only docmap repair must print instructions to remove the created docmap and only remove `.agents/` when the operator confirms it is empty and repair-created. A snapshot-protected repair must print rollback instructions that name the snapshot directory, copied files, target files, and validation command. No rollback command, cleanup command, archive action, VCS action, or lifecycle mutation is implied.

Allowed apply behavior must be idempotent: rerunning the same command against an already-created scaffold, already-created AGENTS contract, already-created docmap, already-created stable spec fixture, already-repaired state frontmatter, or already-repaired docmap returns success, reports no changed paths, and leaves file contents unchanged.
